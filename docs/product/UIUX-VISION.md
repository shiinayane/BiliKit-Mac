# BiliKit App UI/UX 产品蓝图

> 状态：长期产品方向，记录已讨论的信息架构与后续机会，不代表对应服务端能力、持久化 schema 或写操作已经授权。具体能力仍需进入路线图并通过独立契约与 Gate。

## 目标

BiliKit 应成为适合日常使用的原生 Mac 客户端，而不是把网页或当前三栏原型逐项搬进 SwiftUI。界面优先遵循 Mac 的导航、窗口、键盘和辅助功能习惯，同时保留 B 站用户熟悉的视频卡片、播放信息层级和内容语义。

产品形态按“搜索／热门／Library 等一级入口 → 内容页面 → 同窗口独立播放页”组织。侧栏只承担稳定导航；短任务使用 sheet；页面不能依赖假内容、空壳模块或未来接口才能成立。

## 全局导航

- 使用“侧栏 + 主内容”的两栏外壳，搜索位于侧栏最上方，热门与观看历史位于其下，账户入口固定在左下。
- 搜索输入框位于搜索页的主内容 toolbar，不嵌进侧栏。
- 首页、搜索和未来可复用的视频集合使用同一套响应式 16:9 卡片流，但各页面保留自己的加载、空、错误和恢复语义。
- 打开视频进入同一窗口的独立播放页；返回时恢复来源页面、滚动位置和选中卡片。

## 搜索页长期形态

### 有搜索结果时

- 结果直接使用与热门页相同的视频卡片流。
- 分类 Tab、筛选、排序和更多结果类型属于独立搜索功能，不因 UI 精修提前加入。

### 尚未输入关键词时

空页应提供有意义的下一步，而不是永久留出大片空白。候选内容按以下优先级逐步评估：

1. **本机搜索历史**：帮助重复访问；必须支持单项删除与全部清空，并明确只保存在本机。
2. **Bilibili 热搜**：帮助发现当前话题；需要确认稳定的服务端来源、失败语义与匿名／登录边界。
3. **发现内容**：可评估复用现有热门视频，或在服务端能力明确后增加分区入口；不能为了填充版面伪造推荐语义。

这些模块不是必须同时出现。每次只接入能证明用户价值且具有真实数据闭环的最小模块；若内容重复、过时或使搜索意图变模糊，应保持更简单的空状态。

### 搜索历史的数据边界

搜索词可能暴露兴趣、健康、身份或其他敏感意图，不能作为普通界面偏好写入 `UserDefaults`。若立项，应在本地持久化纵向切片中确定：

- 由 Application port 隔离具体存储，Feature 不直接 import SwiftData；
- 容量、保留期限、去重与排序规则；
- 单项删除、全部清空、登出和卸载边界；
- 不保存账号凭据、远端响应正文或不必要的用户身份；
- 是否默认启用及如何向用户说明“仅保存在本机”。

具体使用 SwiftData、文件或其他 adapter，应由持久化契约根据查询和迁移需求决定，不在产品蓝图中预选实现。

## 播放页长期形态

- 主区顺序为“标题—UP 主／播放量／弹幕数—播放器—字幕与弹幕设置—简介—未来评论”。
- 宽屏右栏的长期顺序为“UP 主头像／名称／简介／关注—分 P—相关推荐”；当前没有真实闭环的模块不显示假资料或空壳。
- 只有一个分 P 时隐藏整个分 P 模块；多分 P 时才提供列表与当前项反馈。
- 窄窗口优先保证播放器可观看宽度，侧栏和右栏可以收起；布局变化不得复制或替换播放器 host。

UP 主资料、关注、评论和相关推荐分别涉及新数据或写操作，必须独立立项。确认未来位置不等于授权接口、状态或占位实现。

## 演进规则

- M4.5 只重构已经存在的能力，可实现 1P 隐藏等由现有数据完整支持的呈现规则。
- 新 endpoint、本地内容历史、推荐语义、关注及其他写操作返回对应里程碑单独立项。
- 每个新增模块先回答“它解决什么空白、失败时页面是否仍成立、是否值得增加状态与清理成本”。
- 本蓝图记录方向；`docs/ROADMAP.md` 决定实施顺序，决策契约决定当次授权，验证记录才是完成证据。

## 高刷新率卡片流

2026-07-24 在 4K、最高 240Hz 外接屏上的受控滚动对照表明，当前热门卡片流没有主线程 hang，60Hz 预算充足、120Hz 大体可用，但真实封面与头像纹理的合成成本不足以稳定填满 240Hz。移除渐变、圆角、描边和头像遮罩没有收益；改用占位色后 GPU 中位耗时明显下降；降低图片插值质量同样没有收益。因此 M4.5 不牺牲现有视觉，也不引入通用图片框架或缓存层。

只有后续真实使用仍能稳定复现高刷滚动观感问题，且它的优先级高于功能扩展时，才建立独立、有限的图片呈现性能切片：保持同一卡片与数据契约，仅比较当前 `AsyncImage`、AppKit／`CALayer` layer-backed 图片宿主，以及有证据需要时的预解码缓存。该切片必须同时验证滚动帧成本、图片替换与取消、内存上限、失败占位和离屏清理；不得借机开发通用 benchmark framework。
