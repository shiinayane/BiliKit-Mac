## 隔离上下文审查

三路 reviewer 均使用不继承主对话的全新上下文，并保持工作树只读：

1. `governance_spec` 从附件、AGENTS、Roadmap 和 CI 独立给出最小方案。
2. `governance_audit` 从仓库现状检查流程缺口、冲突和形式主义风险。
3. `governance_diff_review` 只接收 Goal/Context/Constraints/Done when 和实际待审文件，不接收前两路结论，对最终 diff 做 blocker/improvement/reject 审查。

审查发现并完成以下整改：

- **Blocker**：绿区描述可能降低现有 Package 基线；已改为风险只追加验证。
- **Blocker**：公开 API/target/工程的机械变更可能被误归绿区；已明确排除。
- **Blocker**：用户未授权时强制提交/推送/PR 与根规则冲突；已限定为“用户授权 Git 交付时”。
- **Improvement**：本地 `git diff --check` 不覆盖 staged/untracked；脚本增加 cached 检查并明确边界，CI 以 base SHA 检查提交范围。
- **Improvement**：脚本不应覆盖 `xcode-select`；已改为只透传调用方显式设置的 `DEVELOPER_DIR`，缺少完整 Xcode 时给出明确错误。
- **Improvement**：spike 前后不是同一份契约；已拆为 spike 契约和基于测量结果的生产契约。
- **Reject**：逐函数 Changelog、行数硬 Gate、强制真实网络 PR Gate和未经授权的 Git 操作均未采用。

## Gate 发现的既有测试竞态

第一次运行统一 App Gate 时，`SubtitleViewModelTests.newerTrackAndIdentityRejectOlderResultsAndTimeline` 间歇失败：测试在向 `AsyncStream` 发布 timeline 后只执行一次 `Task.yield()`，并不能保证消费者已经处理快照。

生产代码未因此修改。测试改为等待可观察 cue 状态，并把 identity 隔离顺序加强为“当前 identity 命中 → 旧 identity 清空 → 当前 identity 恢复”。目标测试连续运行 10 次均通过。这一结果说明 Gate 不应通过盲目重跑掩盖间歇失败。

## 模型路由试运行

本机 Codex CLI `0.145.0-alpha.30` 的模型目录确认 `gpt-5.6-luna` 可用。先用 Python `tomllib` 解析 5 个项目 TOML 文件，再通过本地 app-server `config/read` 请求传入仓库 `cwd` 和 `includeLayers=true`；返回的有效配置明确包含项目来源、Sol Medium、并发/深度限制，以及 `explorer`、`worker`、`reviewer`、`red_reviewer` 四个角色的绝对配置路径。app-server 使用 `--strict-config` 启动，仓库 trust 状态为 `trusted`。这证明项目层和角色注册已生效；角色真正执行仍须由新任务验证。未受信任的仓库不会加载项目 `.codex/` 配置。

直接让 CLI 模型读取私有仓库 diff 被租户数据策略拒绝，因此没有绕过限制。改用不含项目源码的自包含 Swift 并发题，复现本次真实 Gate 暴露的核心问题：向 `AsyncStream` 发布值后，一次 `Task.yield()` 能否确定性证明消费者已完成处理。Luna Low、Terra Medium 与 Sol High 均正确判断该测试存在竞态，并建议使用带超时的最终状态等待。

这三次调用只比较模型对同一窄问题的推理，不使用项目自定义角色提示，因此不能视为路由试运行。三者均未主动建议升级，而线程/竞态按本项目规则属于红区；若作为 `explorer` 任务，这一点应判为路由失败并升级 `red_reviewer`。项目角色提示已显式补全所有红区触发项，真正的仓库路由仍须从新任务开始验证。

单次运行报告的 token 数分别为 Luna Low 11,581、Terra Medium 2,757、Sol High 2,875。Luna 在这个样本上虽然答对，但 token 反而最多；没有经过相同任务的多轮分布和实际计价，不能宣称已经节省成本。这个样本只证明 Luna 能识别该局部代码问题，不证明它能正确执行项目升级策略，更不支持把复杂 Swift 生命周期、播放或安全审查降级给 Luna。

后续 10–20 个真实任务继续记录角色/模型、有效 blocker、误报、升级、返工和最终 Gate；当前会话的 subagent 运行时不会热加载新项目配置，因此真正的仓库角色路由从新的 Codex 任务开始验证。
