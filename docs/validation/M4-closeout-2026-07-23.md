- 日期：2026-07-23（Asia/Tokyo）
- 系统：macOS 26.5.2（25F84），Apple Silicon `arm64`
- 工具链：Xcode 26.6（17F113）

记录不包含 BVID、CID、标题、字幕／弹幕正文、账号标识、完整 URL 或凭据。本轮只判断既有 M4 是否形成可供 M4.5 使用的稳定基线，不扩展 renderer、播放或业务范围。

## 真实 App

通过已有开发者签名构建启动生产 App，由 Computer Use 执行：

1. 播放中进入和退出 AVPlayer 全屏，时间轴继续推进，字幕与弹幕保持显示；
2. 将窗口从宽布局缩窄后再放宽，播放器未重载，字幕与弹幕保持显示；
3. 分别 seek 到较早和较晚位置，画面、字幕与弹幕按新时间恢复，没有旧内容回流；
4. 播放中替换为第二个视频，旧字幕立即移除；新项目从自己的时间轴开始并正常播放至结束；
5. 切换到搜索入口退出详情，辅助功能树只保留空详情，不再存在播放器、字幕或弹幕控件；
6. 播放期间进程存在一个 `127.0.0.1` loopback listener；退出详情后 `lsof -nP -a -p <pid> -iTCP` 无结果；
7. 同一进程 RSS 从播放中的约 250 MiB 回落到退出详情后的约 224 MiB；随后关闭窗口，仍无 TCP/loopback 句柄，最终退出 App 后进程消失。

真实交互说明当前构建没有出现播放器重启、第二个可见 surface、旧 overlay 回流或 loopback 残留。辅助功能树和进程采样不能直接枚举所有内部 Task、observer 与 layer，因此其归零仍需与确定性测试和生产 probe 合并判断。

## 30 分钟生产 probe

命令：

```sh
DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer \
zsh Scripts/run-m44-renderer-probe.sh 80 1800
```

关键结果：

```text
actual-duration=1800.003 effective-rate=80.000 max-lateness-ms=310.128
emitted=144000 admitted=38410 dropped-no-lane=105590
dropped-capacity=0 peak=140 active-before-stop=133
active-after-stop=0 layers-after-stop=0 root-attached-after-stop=0
rss-peak-mib=95.188 rss-after-stop-mib=80.609
RESULT: PASS
```

RSS 起点为 45.094 MiB；首次预热样本为 95.188 MiB，之后 30 个离散样本在 77.562–83.000 MiB 间波动，末两个样本均为 79.578 MiB。本次执行时输出字段仍名为 `rss-peak-mib` 与 `rss-after-stop-mib`；终审后已分别更名为 `rss-sample-max-mib` 和 `rss-immediate-after-stop-mib`，明确前者只是离散样本最大值，后者只是同步 cleanup 后的即时样本，均不代表采样间真实峰值或延迟回收证明。既有安全 lane 规则按设计丢弃无法安全放置的新事件，没有等待队列；活动对象峰值保持 140，stop 后 controller active count、renderer layer 与 root attachment 归零。

该 probe 只证明 renderer 路径的 RSS 趋势、对象上界和清理；它直接向 presentation controller 注入事件，没有经过 `DanmakuSession`、分段加载或 scheduler，因此不能证明完整生产链路跨分段有界。它也不用于声称 310.128 ms 的单次最大调度延迟代表真实播放帧率。80 events/s 的真实视觉流畅度已在 M4.4 现场验证，本轮真实 App 只验证 resize/fullscreen/seek/替换没有产生可见回归。

## scheduler blocker 修复与完整链路 probe

独立终审发现的无界路径由 D1 窄契约修复：`DanmakuScheduler` 不再为整次播放永久保留所有已投递 ID，而是按现有三段 cache 窗口保留。相同 ID 若连续出现在相邻分段，会在当前分段留下保留别名但不会再次投递，因此旧来源段淘汰时也不会在有效窗口内重复喷射。generation 变化、向后 seek、disable、换视频和 reset 仍整体清空。

确定性测试用虚拟时间跨越八段，覆盖唯一 ID、每段连续出现的相同 ID、相邻边界重复、窗口淘汰和 reset；最终只保留三段、每段四个 ID membership。原有 seek、generation、session 与 presentation 回归合并运行，共 21 项通过。

既有 probe 随后改为经过自制 repository、真实 `DanmakuSession`、scheduler、presentation controller 与 renderer。80 events/s、30 分钟结果：

```text
actual-duration=1800.003 effective-rate=80.000 max-lateness-ms=300.928
emitted=144000 admitted=38415 dropped-no-lane=105585
dropped-capacity=0 peak=140 active-before-stop=135
active-after-stop=0 layers-after-stop=0 root-attached-after-stop=0
segments-requested=6 timeline-subscribers-after-stop=0
session-idle-after-stop=1 rss-sample-max-mib=119.266
rss-immediate-after-stop-mib=77.375
RESULT: PASS
```

32 个离散 RSS 样本从 61.203 MiB 起步，预热后主要在 102.312–119.266 MiB 间波动；后半程没有随第四至第六分段继续单调增长，末样本为 114.406 MiB。该结果与确定性内部计数共同支持“分段数不再造成无界去重增长”，但离散 RSS 不代表采样间真实峰值，也不扩展为 Intel、多窗口或所有真实视频的容量结论。stop 后 session 为 idle、timeline 订阅为 0，controller active count、renderer layer 与 root attachment 均归零。

最终终审后，probe 的 PASS 条件又增加两项低成本防误用：总接纳与丢弃计数必须完整解释目标事件数；当请求时长应跨越三段保留窗口时，实际 distinct 分段数也必须大于三。上述长测原始输出已经满足 `segments-requested=6` 和 `admitted + dropped-no-lane + dropped-capacity = emitted = 144000`；新增字段与断言另以 40 events/s、3 秒 smoke 验证短路径及 stop 清理，短路径明确输出 `retention-window-crossed=0`，不冒充跨段证据。

## 独立红区终审

首次终审发现一个 blocker：`DanmakuScheduler.deliveredIDs` 会记录正常播放中每个已投递 ID，只在 reset、generation 变化或倒退 seek 时整体清空；三段 cache 淘汰不会裁剪它。连续无 seek 播放因此存在随分段增长的明确生产路径，而 renderer-only probe 绕过了该路径。

该发现曾推翻“本机候选通过”，但不推翻真实 resize/fullscreen/seek/替换、renderer 有界和 loopback 清理这些局部证据。D1 修复及完整链路本机验证完成后，由未参与实现、且不继承实现推理的新上下文进行第二次红区终审：blocker 为 none，结论为“不阻断提交 PR，但不等于 M4 Gate 已关闭”。

终审提出的两项 improvement 已整改：probe 对长参数自行要求跨越保留窗口并核对事件总账；scheduler 测试增加换 identity 后同 ID 可重新投递，以及 disable→enable 不回填禁用区间。整改后的 scheduler 六项测试、短 probe 与 179 项统一 App Gate 均通过；最终复核无新增 blocker。

## 当前结论与未覆盖边界

M4 暂不能正式关闭。本机候选证据现包括真实 resize/fullscreen/seek/替换和退出详情、可观察的 loopback 清理、跨六段完整生产链路的 30 分钟 RSS、scheduler 内部有界计数、stop 后 session/timeline/renderer 清理、最终独立红区终审，以及修改后的统一 App Gate。

以下内容不由本机记录替代：

- 当前分支仍需在 PR 上通过 macOS 15/26 CI，才能把 M4 总 Gate 标记为正式关闭；
- 本轮不是 Intel、更多真实视频、所有显示设置或多窗口容量证明；
- M4.5 若改变播放器结构、route owner 或 surface 生命周期，必须复用本记录作回归基线，不能把 UI 重构后的新行为反写为 M4 已验证事实。
