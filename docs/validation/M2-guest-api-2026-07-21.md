# M2 游客 API 核心链路验证——2026-07-21

## 结论

`BiliAPI` 数据层已具备接入 App 的最小游客链路：热门或搜索 → 视频详情 → 分 P → AVC/AAC 播放描述。匿名 WBI key 获取、签名、搜索解码和单次 key 刷新重试已由受控测试覆盖，并在真实网络中通过。

App composition root、游客主流程状态模型和最小三栏 `NavigationSplitView` 已于同日接入。真实界面已经完成热门、WBI 搜索、详情/首分 P 与系统播放器的纵向验证；热门/搜索切换、旧请求隔离和按原请求重试也已由受控测试固定。M2 Gate 关闭。

## App 真实界面证据

- 冷启动后匿名热门页加载 20 条当前视频，列表显示标题、UP 主、播放量、时长和 CDN 封面。
- 选择第一条热门视频后，详情区显示简介、分 P 列表与首个分 P 的 `AVPlayerView`。
- 本次样本的播放器时长为 `29:13`；点击播放后时间轴从 0 秒推进到 11 秒，随后可正常暂停。
- 视觉检查曾捕获 HTTP/协议相对封面地址在 ATS 下退化为占位图的问题；图片地址统一规范化为 HTTPS 后，完整退出旧进程并重新启动最新构建，确认封面实际显示。
- 在搜索入口提交 `macOS` 后，真实接口返回 19 条可打开结果、服务端总数约 1,000；搜索封面完成异步加载。
- 选择首条搜索结果后成功进入详情与 P1，播放器时长为 `22:27`，时间轴从 0 推进到 10 秒以上。
- 播放期间切回热门会清空选择、移除播放器并暂停；再次进入搜索会保留关键词并重新加载结果。
- 实际界面检查发现“提交搜索后立即切换入口”存在两个 UI Task 争抢最后状态的风险，随后统一为单个 `task(id:)` feed 意图，最后一次界面意图成为唯一有效请求。
- UI smoke test 已修正宿主 target 名并能随工程编译。本机执行时 Xcode UI test worker 在启动断言前持续等待，因此该项不计为运行通过，也不作为本次纵向链路证据。

## 边界与后续工作

- 真实搜索的 20 条原始 `video` 结果中，有 1 条没有可用 BVID。数据层会过滤这类无法打开的条目，但 `totalResults` 继续保留服务端原始计数，因此页内数量可能小于 `pageSize`。
- 搜索结果和 WBI key 是动态外部依赖；线上单次失败不能单独证明本地签名或解码回归。
- 当前没有对并发的首次搜索合并 nav 请求；UI 只在显式提交时搜索，并通过 feed 代次取消和隔离旧请求。
- UI smoke test 的本机 worker 启动问题不影响 App target 构建、状态模型回归或本次真实界面证据，但后续仍应在可稳定启动 XCUI worker 的环境复核。
