## 生命周期与负向证据

- `playerContent()` 在生产 View 树中只有一个声明点；响应式 mode 只增减
  compact disclosure 或 wide rail sibling。
- 多分 P 假值路径实际经历 1080×680 → 1320×820 → 1080×680。切换前后
  host create event、playback load identity、subtitle track identity 与
  danmaku start identity 均未增加，`active == 1`、`peak == 1`。
- 字幕基线必须与唯一 playback identity 完全一致，避免空数组与空数组相等
  造成假通过。
- 返回 A 后 host identity 成对 dismantle 且 `stopCount == 1`；播放 B 可以
  创建新 host，关闭后第二对 identity 成对 dismantle，最终
  `active == 0`、`stopCount == 2`、全序列 `peak == 1`。
- 全屏内部 AVKit host identity 没有可用脱敏 hook，本记录只声称可见播放
  连续，不把人工观察冒充对象级证明。

## 签名 App 观察

使用本机现有开发签名构建 Debug App，并以无个人数据记录的公开样本完成：

1. 单分 P 页面暴露 `playback.layout.single`，没有“分 P”标题、入口、右栏
   或空白占位；主区自然占满可用 detail 宽度。
2. 多分 P 在带 sidebar 的紧凑宽度显示原生 disclosure；增加 detail 可用
   宽度后显示固定 400 pt 只读右栏，当前项、标题与跨小时时长可见；恢复
   sidebar 后回到 disclosure。
3. 播放器始终位于黑色 16:9 容器内，标题、播放器、控制与简介共享
   40 pt 水平／24 pt 垂直阅读边界；左侧 detail 背景清楚可见，没有贴住
   split view 分隔线。B3 签名 App 复查确认 400 pt 右栏，以及标题保持
   `title`、元信息使用 `body`、其余页面文字使用 `title3` 后，没有挤压
   播放器、造成控制不可达或改变画面字幕字号。
4. 播放时间在 wide → compact 与播放中进入／退出 AVKit 全屏期间持续递增，
   没有回到零；字幕和弹幕控制保持可达。
5. 一个 9:16 样本完整居中显示，左右使用黑色 pillarbox，没有
   aspect-fill 裁切。
6. A 返回后进入 B，再关闭窗口，来源、播放和清理均可到达。

这些观察证明真实 `AVPlayerView` 的可见布局与连续性，不证明服务端长期
可用性、所有视频比例、Intel、辅助显示矩阵或全屏内部对象 identity。
