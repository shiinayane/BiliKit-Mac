# M4.5 Slice B 播放页响应式重排技术草案

> Revision：BT3。状态：B1 已由用户于 2026-07-24 确认，B3 绑定快速视觉调整最终确认的 40 pt 水平边距、24 pt 垂直边距、400 pt 右栏及页面文字层级；本文记录最小实施方法，不要求用户逐项确认技术细节。决策范围以 [`M4.5-playback-layout-decision.md`](./M4.5-playback-layout-decision.md) 为准。

## 当前事实

- `AppShellView.PlaybackPageRoot` 只组合现有 `VideoDetailColumn`、返回 toolbar 和稳定的 `playerContent` 值。
- `VideoDetailColumn` 只根据 `GuestVideoViewModel.State` 选择 idle／loading／detail／failure，并以 `PlaybackItemIdentity` 驱动字幕和弹幕。
- `GuestVideoDetailView` 当前只声明一次 `playerContent()`，但采用“播放器 → 单列 ScrollView”的验证布局，分 P 无条件显示。
- `PlayerHostLifecycleProbe` 已在 `NSViewRepresentable.makeNSView`／`dismantleNSView` 边界记录 active、peak 与 `ObjectIdentifier`；Slice A 已固定 A→返回→B 与关闭窗口的清理。

## Goal / Constraints / Done when

**Goal**：在不改变 App route 或播放会话的前提下，把 ready／preparing detail 重排为宽窄响应式播放页，并隐藏单分 P 模块。

**Constraints**：

- `playerContent()` 在生产 View 树中只出现一次，不能放进两个完整布局分支。
- 不修改 `GuestVideoViewModel`、Application port、playback engine、renderer、subtitle／danmaku owner 或 Xcode／Package 配置。
- 分 P 继续只读；零分 P 继续走既有 adapter／ViewModel 失败路径。
- 响应式状态不持久化，不进入业务 ViewModel，不触发播放副作用。

**Done when**：B1 的产品呈现、单 host／identity、清理、App Gate、签名 App 关键路径与红区审查证据全部成立。

## 最小实现结构

在 `GuestVideoDetailView` 内保留一棵稳定主区：

```text
available width
└── HStack
    ├── main scroll
    │   ├── title + metadata
    │   ├── 16:9 black container
    │   │   └── playerContent()    # 唯一声明点
    │   ├── subtitle controls
    │   ├── danmaku controls
    │   ├── compact parts disclosure（仅多分 P 且窄）
    │   └── summary
    └── wide parts rail（仅多分 P 且宽）
```

使用最外层、尚未加入 rail 前的播放页局部可用宽度，通过一个无副作用纯函数推导私有 `PlaybackLayoutMode`：

- `singlePart`：`pages.count == 1`，不显示任何分 P UI；
- `compactParts`：多分 P、宽度不足，主区内显示原生 `DisclosureGroup`；
- `wideParts`：多分 P、宽度足够，主区右侧显示固定 400 pt 的独立滚动栏。

空 pages 不对应任何布局 mode；`GuestVideoContext` 的既有构造路径必须先失败关闭。不能从已经被 rail 压缩的 main scroll 宽度反推 mode，否则阈值附近会在 compact／wide 间振荡。阈值只负责保护主观看区的最小可读宽度加右栏，不表达设备类别。实现时从最外层实际 detail 可用宽度选择一个值，使 1080×680 基线进入 `compactParts`、1320×820 基线进入 `wideParts`；若不能同时成立，先调整局部 metrics，不引入第二断点或 hysteresis。

main scroll 使用文件内私有的 40 pt 水平 padding 与 24 pt 垂直 padding，让标题、播放器、控制和简介共享阅读边界，并确保播放器左侧不会贴住 split view 分隔线。这组值只属于当前播放页，不复用视频卡片的 `VideoCardGridLayout`，也不建立全局 spacing token。标题保持 `title`；元信息使用 `body`，简介、分 P 和字幕／弹幕控制使用 `title3`；画面字幕不跟随页面字号。

播放器外层按主区扣除 padding 后的可用宽度声明 `.aspectRatio(16.0 / 9.0, contentMode: .fit)` 和黑色背景；不继续使用当前独立的 min／ideal／max height 拉伸约束。`AVPlayerView` 仍负责非 16:9 内容的等比适配，SwiftUI 不裁切媒体画面。padding 和 aspect-ratio modifier 只能包在稳定主区的同一播放器节点外，不能通过 mode 分支重新包装 host。

`playerContent()` 始终位于 main scroll 的同一结构位置。mode 变化只切换分 P disclosure／rail 这两个 sibling，不对播放器或主区使用 `.id`、`AnyView` 重新包装、matched geometry 或 transition。

## 分 P 呈现

把当前分 P 行提取为文件内私有只读 View 或 `@ViewBuilder`：

- 当前项使用现有 `play.circle.fill` 与 accent color；
- 非当前项使用次要样式；
- 标题最多两行，时长保持 monospaced digit；
- 整行不加 Button、tap gesture、selection binding 或 focus action；
- compact 与 wide 复用同一个行实现，不复制格式化规则。

`pages.count == 1` 不保留“分 P”标题、divider、disclosure 或空白。`pages.isEmpty` 不得进入 `singlePart` 或在 View 中制造选中项；既有 `GuestVideoContext` 构造失败继续负责 fail closed。

## 自动化证据

1. **Feature 布局策略测试**
   - 以纯函数固定零分 P 不进入布局、单分 P 不产生辅助区、多分 P 在 detail 可用宽度 870 pt／1100 pt 分别进入 compact／wide。
   - 固定 40 pt 水平／24 pt 垂直内容边距、400 pt 右栏与 16:9 尺寸计算；真实 View 的四边边距、文字层级、分 P 文案、当前项和跨小时时长由签名 App 检查，不用不稳定的 SwiftUI accessibility 树冒充确定性单测。
   - preparing 与 ready 复用同一生产布局节点，准备遮罩不改变页面结构。

2. **App host 生命周期测试**
   - 为 Slice B 用例参数化现有假仓库，使播放 A 返回至少两个手写分 P；不能复用只返回一个 page 的 Slice A 路径冒充响应式证据。
   - 在同一播放 route 内把附着窗口的 hosting view 从 1080×680 改为 1320×820 再改回。纯函数先固定这两个 detail 宽度对应 compact／wide；实际 resize 则等待播放器 frame 稳定为 16:9，并以加入固定右栏后主区宽度收窄、切回后恢复原宽证明布局已经完成，避免并行测试中不稳定的 SwiftUI accessibility 树。
   - 在第一次 compact 几何和 `active == 1` 成立后记录唯一 created identity、event count、playback load identity、subtitle tracks identity 与 danmaku start identity。两次 mode 转换后断言这些 identity／次数没有增加，`active == 1`、`peak == 1`。
   - 返回必须产生与 A created identity 配对的 dismantled event。播放 B 可以 create 新 host 与新 playback identity；关闭必须产生与 B created identity 配对的 dismantled event，最终 `active == 0`，全序列 `peak == 1`。
   - 最小调用记录只加在现有 `SuspendingPlayback`、subtitle 假 repository 和 recording presentation；不增加生产 probe、第二计数器或通用事件总线。没有直接公开的 generation 不做对象级声称；“resize 未触发新的 select/load”作为本切片的行为证据。
   - 自动化不把 AVKit 原生全屏内部 host identity 声明为已证明。签名 App 关键路径只记录用户可见播放是否连续、是否重载；若观察到异常，再决定是否能以现有脱敏 hook 增加定点证据。

3. **回归**
   - 既有 route、A→返回→B、播放失败／重试、字幕／弹幕 identity、窗口关闭与 Slice A UI smoke 保持通过。
   - 最终只运行最高适用的 `BILIKIT_COMPACT_LOGS=1 sh Scripts/run-quality-gates.sh app`。

## 真实环境与审查

- 签名 App 使用无敏感输出的真实播放路径，检查 1080×680、1320×820、自由缩放与 AVKit 全屏；确认播放器保持 16:9、周围页面背景清楚可见且没有贴住 split view 分隔线。至少观察一个非 16:9 视频，确认完整画面以 letterbox／pillarbox 呈现而非 aspect-fill 裁切。它证明可见连续性与交互结果，不单独证明内部 host identity。不记录标题、BVID、字幕／弹幕正文、画面或完整 URL。
- 用户只需确认：播放未因宽窄／全屏重启，单分 P 没有空模块，多分 P 宽窄位置合理，返回来源正确。
- 实现后由独立 `red_reviewer` 检查 View identity、`NSViewRepresentable` 生命周期、播放 item／generation、字幕／弹幕 task、返回与关闭清理。
- 只有 engine／renderer 被触碰或关键序列出现资源趋势异常时才重跑 M4 的 renderer／30 分钟 probe。

## 停止与回退

- 如果稳定主区仍会在 mode 变化时 dismantle player host，停止并恢复 Slice A 详情布局；不通过保留隐藏的第二 host 或缓存 NSView 绕过。
- 如果 SwiftUI 局部宽度无法在两个基线间形成可靠响应式行为，只比较“接受始终 compact”与“一个最小 AppKit size observation adapter”两个低成本回退，先回到决策 Gate。
- 如果实现需要改变 `AppNavigationModel`、`ContentView` 清理顺序、播放器 factory、Feature 依赖方向或分 P 数据能力，停止并重新请求授权。
