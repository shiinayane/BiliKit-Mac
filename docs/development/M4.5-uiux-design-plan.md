# M4.5 UI/UX 设计与实施草案

> Revision：UT6。状态：设计草案，不代表用户逐项确认；可在不改变 U6 的授权范围、页面结构和复杂度预算时调整。当前不授权生产 Swift 实现。

## 已核对的当前事实

- `ContentView` 通过工具栏打开历史和账户 sheet。
- `BrowseNavigationView` 使用三栏 `NavigationSplitView`：入口、固定宽度 Feed、详情播放器。
- 热门和搜索以 128×72 封面的行列表呈现。
- 播放详情把播放器、标题、简介、字幕、弹幕和分 P 放在同一纵向表面。
- 当前 Domain、Application 与 API 没有评论和相关推荐模型或 port。

## 外部体验采样

- 2026-07-23 的 B 站网页首页采用封面优先的响应式视频卡片；播放页采用主观看区与较窄右栏。登录、地区、窗口和 A/B 实验可能改变具体内容，因此只采纳信息层级，不复制像素、素材或运营入口。
- Apple TV for Mac 使用 sidebar 组织一级入口，并把账户入口放在侧栏底部。BiliKit 只迁移导航习惯，不迁移订阅、购买或多 profile 语义。
- BiliKit 保持非官方身份，使用系统控件与自己的视觉语言，不复制 B 站商标外观或网页组件实现。

## 建议路由

App shell 持有类型化 route，不把具体 ViewModel 或 Repository 跨 Feature 传递：

```text
AppRoute
├── browse(.home)
├── browse(.search)
├── library(.history)
└── playback(bvid)
```

- 首页／搜索卡片与历史条目只发送播放 intent。
- App shell 负责把 intent 路由到 `BiliBrowseFeature` 的播放页；Composition 只组装依赖。
- 返回快照只包含来源 route、搜索 query、滚动 anchor 与选中 BVID，只在窗口内存中保留；登出或窗口关闭时清除含历史来源的个性化快照。
- 播放页消失、切换视频或登出继续调用既有 reset／cancel／资源清理路径。

最终公开类型数量和 owner 需在生产契约中固定；本草案不预先增加公共 API。

## 表面拆分

### App shell

- 将三栏导航改为两栏 `NavigationSplitView`。
- sidebar 第一项为搜索 route，其下为热门与观看历史；底部为账户状态按钮。
- 选中搜索 route 时使用系统 toolbar search placement 呈现输入框；侧栏只显示“搜索”入口，不承载输入控件。
- 历史变为主内容 route；认证继续使用短生命周期 sheet。
- 侧栏收起状态由系统行为驱动，不自制浮层导航。

### 卡片流

- 首页与搜索复用同一窄职责视频卡片视图，但各自保留页面状态和错误语义。
- 使用 `LazyVGrid` 与自适应列宽；封面固定 16:9，标题两行，UP 与播放量一行，时长叠在封面角落。
- 卡片必须有 hover、pressed、selection、focus 和键盘激活反馈；Reduce Motion 下关闭非必要缩放或位移动画。
- 图片失败使用稳定占位，不改变卡片尺寸。

### 搜索页

- sidebar 的“搜索”是页面选择；输入框位于主内容顶部 toolbar，并保持当前 query。
- 搜索结果从第一项开始直接使用与热门页相同的响应式 16:9 视频卡片流，不增加“最佳匹配”分组，也不照搬 Apple TV 的竖版影视海报。
- 分类 Tab、筛选、排序与更多搜索类型属于后续功能；本次不增加控件、占位 View、页面状态或公共接口。
- 清空输入回到搜索空状态；返回热门再回来时，窗口生命周期内保留 query 和已有结果。
- 搜索空页在 M4.5 只提供简洁引导。搜索历史需要持久化与隐私契约，Bilibili 热搜需要服务端 contract，热门／分区发现需要单独确认产品价值与状态 owner；均记录在 App 级 UI/UX 蓝图，不提前混入本次实现。

### 播放页

- 宽屏：可调整的主观看区 + 右栏；右栏先承载现有分 P。
- 窄屏：系统可收起 sidebar，右栏折叠成可展开区域；播放器维持可观看宽度。
- 主区按“标题—UP 主／播放量／弹幕数—播放器—字幕与弹幕设置—简介”组织。标题与元信息形成同一信息组，字幕和弹幕设置紧贴播放器，简介位于控制之后。
- 未来右栏顺序固定为“UP 主头像／名称／简介／关注—分 P—相关推荐”。本次右栏仍从已有分 P 开始，不为尚未接入的 UP 主资料、关注或相关推荐创建 View、state、公开 API 或空白占位。
- `context.pages.count == 1` 时不创建分 P 表面；只有 `count > 1` 时显示现有分 P 列表。零项仍由既有无效响应路径处理，不能伪装成单分 P。
- 不把播放器放入会随正文滚出视野的嵌套列表；具体 sticky 行为在真实原型中比较后决定。
- 播放 route 内只声明一个具有稳定 identity 的 player host；宽窄变化只改变它的兄弟右栏，不在条件分支中移动、复制或替换 host。

## 最小视觉语言

- 优先使用系统背景、sidebar material、selection、control 与 focus ring。
- 首轮 token 仅限重复出现的页面边距、卡片间距、卡片圆角、封面比例、正文最大阅读宽度和弱化文本层级。
- 不先建公共主题 target；若 token 只被一个 Feature 使用，留在 owner 内部。
- 品牌强调色必须在浅色、深色和 Increase Contrast 下可辨识，且不能成为唯一状态信号。

## 分阶段实现建议

### Phase 0：静态原型

- 用自制假值 Preview 验证首页、搜索、历史、播放页在 1080×680 和 1320×820 的布局。
- 验证侧栏／右栏收起、文字放大、深浅色和 Reduce Motion。
- 只确定导航与信息层级，不接真实评论或相关推荐。

### Phase 1：导航与首页

- 固定类型化 route、两栏 shell、账户底部入口、历史主页面和卡片流。
- 保持已有 Use Case、取消与错误映射不变。
- 固定返回后的内存级浏览上下文。

### Phase 2：播放页

- 迁移现有播放器、详情、分 P、字幕和弹幕控件。
- 用单分 P 与多分 P 两个确定性状态固定模块显隐；显隐不得触发播放项目重载或改变所选 CID。
- 验证播放不中断、seek／倍速／切分 P／登出清理无回归。
- 宽窄窗口只切换版式，不创建第二套播放器 owner。

### Phase 3：精修与 Gate

- 统一 loading、empty、error、retry、hover、focus 和 selection。
- 补齐 M4.5 主要控件的键盘、VoiceOver 标签／顺序／可达性，以及 Increase Contrast、Reduce Transparency、Reduce Motion；完整 VoiceOver 导航仍属于 M5。
- 使用无个人数据截图矩阵和 M4 性能对照关闭 Gate。

## 生产实现前必须补齐

- 先关闭 M4 总 Gate，记录既有 resize、fullscreen、登出清理、资源归零和性能基线；不为 M4.5 扩张成新矩阵。
- route 与滚动恢复的唯一 App shell owner、不变量和失败恢复。
- 一张最小 transition 表：离开播放页、切视频、快速返回后再选择、登出／认证失效、离开历史和关闭窗口；固定每个事件恰好一次调用的既有 reset／cancel，以及个性化快照清理。
- sidebar／右栏响应式断点采用容器行为还是测量值；不得只凭单一设备硬编码。
- 播放器在布局切换时保持同一 engine 与稳定 host 的证明：跨宽窄、全屏和返回时并发 host／surface owner 峰值为 1，无 item reload 或时间轴重启，退出后 host、surface、renderer、observer 与 loopback 归零。
- History 从 sheet 迁移后，关闭、登出、重新认证和分页任务的清理路径。
- Preview 假值、XCUI smoke、无障碍和视觉截图矩阵。
- 未参与实现的新上下文分别做可理解性审查与线程／资源终审。

验证预算只覆盖 route／取消／generation／登出的负向自动化、两个既定窗口、一次自由跨宽窄／全屏关键路径，以及复用 M4 soak；视觉和辅助功能取成对代表样本，不做完整笛卡尔积。若稳定单一 player host 无法成立，回退为固定主观看区与可收起右栏，不改 engine／renderer 生命周期路线。
