# M4.5 导航与窗口 Chrome 架构决策草案

> Revision：A1。状态：用户已于 2026-07-24 明确确认三项授权；职责迁移已实现并通过价值／简化、可理解性、技术、红区终审与完整 App Gate，当前只等待用户确认迁移后观感。M4.5 其余视觉实现仍冻结在 Slice A 提交 `0b1f94a`。`chrome` 在本文只指窗口标题、页面标题、toolbar、侧栏和返回入口，不指网页浏览器内核。

## 为什么现在停下来

Slice A 已形成可用的两栏导航，但 `ContentView` 同时负责依赖组装、窗口生命周期、一级路由、侧栏、四类页面创建、搜索请求、认证协调和页面 toolbar；`WatchHistoryView` 又在 Feature 内自行注入 toolbar。于是“热门和搜索不显示标题，历史显示标题并在右侧刷新”这样的页面规则，需要同时猜测 Scene、App shell 和 Feature 的行为。

这不是继续微调 modifier 可以解决的问题。若在现状上进入播放页重排，页面 chrome、播放生命周期和跨 Feature 组装会继续聚集在同一个文件。

## 本轮只支持两个决策

1. **职责边界**：窗口、App 导航、页面 chrome 和 Feature 内容分别由谁拥有。
2. **迁移方法**：怎样在不改变现有路由、搜索、认证和单播放器生命周期的前提下，把职责拆开。

本轮不决定视觉像素、播放页新布局、通用设计系统、跨平台抽象或新业务能力。

## 候选与结论

### 方案 A：继续由 `ContentView` 集中配置所有页面

优点是文件少；缺点是新增页面会继续增加 route 分支、toolbar、任务和生命周期交叉。它正是当前问题，淘汰。

### 方案 B：一个最小 App shell 内建立页面 chrome 边界（推荐）

- **Scene**：只设置窗口默认尺寸和统一 toolbar 外观，不判断当前 route。
- **`ContentView`**：继续持有窗口级 UI 状态与模型，并且是现有顶层 `.task`、`onChange`、`onDisappear` 与跨页面／窗口清理的唯一 owner；具体 adapter 仍只由 Composition 组装，本轮不移动这些副作用。
- **一个私有 App shell 文件**：容纳 `NavigationSplitView`、侧栏、route switch，以及同文件内的四个私有页面根。页面根只声明语义标题、标题是否在 toolbar 显示和该页 toolbar item。
- **Feature 内容**：显示业务状态并发送最窄用户意图；不决定窗口样式、侧栏或其他页面的 chrome。

热门和搜索保留语义 `navigationTitle`，但页面根用 macOS 15 的 `toolbar(removing: .title)` 隐藏默认可见标题；历史页保留可见标题，刷新在常规窗口进入右侧操作区，窄窗口允许系统按原生规则收纳；搜索页的输入框使用 `.principal`；播放页的返回使用 `.navigation`。Scene 不再用全局 `hiddenTitleBar` 代替页面决策。

### 方案 C：把 route host、页面 chrome 拆成多个独立文件

边界与方案 B 相同，但预先增加 `AppRouteHost`、`AppPageChrome` 等文件。当前页面和 chrome 规则很少，拆分本身不增加正确性；只有单文件确实再次混合职责时才考虑，首轮淘汰。

### 方案 D：建立通用 `PageChromeDescriptor`／toolbar registry

它能把配置集中成数据，但 SwiftUI toolbar 是泛型 View，最终容易引入类型擦除、环境注册或命令总线。当前只有四个页面，没有证据需要框架，淘汰。

## 必须保持的边界

- `AppNavigationModel` 继续是唯一一级 route 与播放开始／停止副作用 owner；本轮不重写路由模型。
- 一个窗口仍只有一个 player host、subtitle owner 和 danmaku owner；拆 View 不得改变其 identity 或清理顺序。
- `playerContent` 继续只在 `ContentView.init` 构造一次并作为稳定值传入 shell；不得在 `body`／shell 中再次调用 factory，也不得用 route 或 BVID 的 `.id` 强制刷新播放子树。
- Feature 之间不互相 import；App／Composition 仍是跨 Feature 组装处。
- 搜索、热门、历史、认证和返回的现有可观察行为不变。
- `navigationTitle` 是页面的语义身份，不用空字符串冒充隐藏策略；视觉显隐由页面 chrome 决定。

## 复杂度预算

- 不新增 target、Package、第三方依赖、环境命令总线、toolbar registry 或通用页面协议。
- 生产迁移最多新增一个 App 文件，暂称 `AppShellView.swift`；页面根先作为同文件私有类型。
- `ContentView` 保留依赖生命周期、现有顶层任务与清理；新文件必须移走 split view、sidebar、route switch 和页面 chrome，禁止只包一层转发。
- 首轮只验证三个页面规则：热门隐藏标题、搜索隐藏标题且输入框居中、历史显示标题且刷新在常规窗口位于右侧操作区；窄窗口只要求仍可发现和使用。
- 若需要移动现有任务／清理 owner、修改 `AppNavigationModel` 状态机、播放／字幕／弹幕实现，或新增超过上述预算的抽象，立即停止并重新评估。

## 最低证据与停止条件

完成架构迁移至少需要：

- 三个页面规则在 macOS 15 构建和当前签名 App 的默认／窄窗口中成立，切换页面后没有遗留、重复或错位 toolbar item；历史加载或分页时刷新仍禁用。
- 现有 route、搜索、历史离开／返回、认证失效、A→返回→B 和窗口关闭测试保持通过；一条“播放 A→返回→再播放 B→播放中关闭窗口”的真实 App route 测试同时证明两个播放期间 host 没有额外 create/dismantle，返回与播放中关窗各只清理当前 host。
- `ContentView` 不再直接构造每个页面的 toolbar；Feature 内容不再决定窗口级 chrome。
- App Gate、无个人数据 UI smoke、技术审查和红区生命周期审查通过。

如果 SwiftUI 的页面级标题移除在支持系统上表现不稳定，先停止；只比较“最小 AppKit toolbar adapter”和“接受统一可见标题”两个低成本回退，不建立自制窗口框架。实施前由用户确认本轮职责边界、成本与回退原则；迁移完成后用户只需确认三个页面的真实观感。两次确认都通过后，才解冻 M4.5 并重新决定是否进入播放页重排。
