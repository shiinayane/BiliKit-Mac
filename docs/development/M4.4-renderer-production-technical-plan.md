# M4.4 renderer 生产技术实施计划

> Revision：PT4。状态：P4 已由用户于 2026-07-23 的真实播放验证和镜像覆盖澄清明确授权。PT4 只纠正覆盖比例与纵向锚定模型，不增加显示区域/密度 UI、第二 renderer 或通用框架。

## 事实输入

- M4.3 已有 `@MainActor DanmakuSession`、`DanmakuScheduler`、`DanmakuBatch`、统一 `PlaybackTimelineProviding`、generation、两请求/三段缓存和 `bufferingNewest(4)`。
- spike 证明原始 A/B 的共同失败来自逐 layer compositor shadow，不是 CATextLayer 本身；内容级 1.5 pt 重墨在当前 Mac 的 40/80 events/s 各两次运行中保持现场流畅，内存和 319/639 个活动 layer 稳定，最终 owner 释放通过。
- app-side `CADisplayLink` 不能代替实际 presentation 证据。快速 probe 记录计数和内存，现场流畅度与必要时的 Animation Hitches 另行判断。

## 拟议所有权

### 平台无关调度层

- 保留 `DanmakuScheduler` 对过滤、去重、时间窗和 discontinuity 的现有职责。
- 新增纯值类型的呈现配置、surface 尺寸、呈现项和 renderer 统计；不得包含 AppKit/Core Animation 类型。
- lane allocator 使用虚拟时间和几何输入，独立计算 scrolling/top/bottom lane、碰撞和丢弃，可在无窗口测试中确定性验证。

### `@MainActor` 单序列呈现层

- 固定使用一个 `DanmakuPresentationUpdate`：同一次 MainActor timeline 回调内携带 snapshot、identity、discontinuity generation、可选 batch 和 clear 语义。pause、rate、seek/clear 即使没有新事件也必须产生 update。
- `DanmakuSession` 同步调用一个弱持有的纯值 presentation sink；生产 renderer 不消费现有带 `bufferingNewest(4)` 的独立 batch stream。该 stream 可保留给现有观察者，但不能驱动产品呈现。
- presentation owner 顺序处理 update，以 identity + discontinuity generation 拒绝旧输入，并把纯值命令交给一个窄 renderer 协议。
- 同一 MainActor transition 必须先让 backend 移除 allocator 返回的 `expired/drained`，再为本轮 `admitted` 创建对象；只有 backend 已同步执行全量 `clearAll` 时才允许显式忽略 drain 列表。
- Core Animation backend 另有本地 render epoch。任何 clear、resize、surface attach/detach、总开关关闭、stop/close 都推进 epoch；animation completion 携带 `epoch + object identity`，relay 弱引用 owner，失配只 no-op。
- “呈现命令协议”和“AppKit 挂载 adapter”分开：前者只接收纯值；Platform 中的具体 overlay host 可以知道 Core Animation backend 的 root layer，并负责 attach/detach。未来 backend 替换对应 adapter，不预建统一 Metal/AppKit surface。
- 每个 `WindowGroup` 实例独立创建 session、presentation controller 和 renderer，不在进程级静态环境中共享可变呈现状态。
- overlay host 使用唯一 owner token 挂载、更新和卸载 surface；旧 host 的迟到 resize/detach 不能清空或移除新 host。首次布局即使为 0×0 也必须传递，以关闭 admission 并清空旧几何。
- 生产 overlay 挂载到 `AVPlayerView.contentOverlayView`，使其跟随 AVKit 的播放器内容 surface 与 fullscreen 转换；不以播放器外部的 SwiftUI sibling 充当 danmaku surface。

## 样式与动画

- 默认 24 pt system semibold，使用系统 fallback 覆盖中文、日文、韩文、拉丁和 emoji。
- attributed string 使用白色 fill 与内容级黑色 `NSShadow`：零 offset、1.5 pt blur。承载 layer 的 `shadowOpacity` 必须为 0；不得同时使用 fill+stroke。
- 每条滚动弹幕使用一次线性 Core Animation position animation；不由 display-link 每帧改 position。
- pause/rate 通过 layer local time（`speed`、`timeOffset`、`beginTime`）或等价可测试机制同步；seek/discontinuity 直接清空并建立新 generation，不尝试续接旧 animation。
- 顶部/底部弹幕使用有界的固定展示时间；具体滚动/固定时长与基础字号常量集中定义，不散落在 view 或 animation delegate。

## lane、容量与丢弃

- lane 高度由字体实际 bounds、最小垂直间距和 surface scale 推导；surface resize 时推进 render epoch，废弃旧 lane 状态并清空活动内容。
- scrolling lane 只有在前一条弹幕的尾部已进入安全距离，且按两条速度预测不会追尾时才可复用。
- overlay 只有一个共享覆盖比例。top 与 scrolling 从 surface 上缘向下分配，bottom 从 surface 下缘向上镜像分配；三种 mode 分别维护自己的 lane 占用，因此上下范围相交时跨类型可以落在同一位置，当前默认只阻止同类型碰撞。
- 覆盖比例当前固定为 100%。未来 10%/25%/50%/75%/100% 改变上下各自从边缘覆盖的高度：50% 的上下范围在中线相接，75% 在中间 50% 相交，100% 完全重叠。未来“正常／较多／重叠”改变同类型 admission 策略。PT4 不预建这些设置或策略枚举。
- 一批事件保持 scheduler 的时间与 ID 稳定顺序。已显示对象不被抢占；无安全 lane 或达到 640 active hard cap 时丢弃当前新事件。
- 单次 MainActor update 最多尝试 640 个新事件，超过部分按容量丢弃，避免恶意同时间戳 burst 在 admission 前制造无界 Core Text 工作；Core Animation backend 自身在创建对象前再次执行 640 hard cap。
- 正文超过 512 UTF-16 code units 时在创建 attributed string/Core Text 对象前 fail closed；测量结果另受 8192×512 pixel 上限保护。两者是本地资源边界，不作为用户可调样式或 renderer 胜负指标。
- 不建立等待队列。统计只保留 admitted、droppedNoLane、droppedCapacity、active 和 peakActive 计数，不记录正文、事件 ID、视频标识或远端字段。
- PT4 不实现 layer pool 或文字 cache。若真实 probe 证明 allocation 是主要瓶颈，必须先提供分配证据和新的保留上限。
- 总开关关闭会同步 clear 并推进 render epoch；三类过滤变化清空当前内容并按新过滤继续，不回填。P4 不包含其他几何或视觉偏好 UI。

## 实施切片

1. **纯算法切片**：呈现值类型、lane allocator、碰撞预测、容量与稳定丢弃顺序；只改 `BiliDanmaku` 和测试。
2. **renderer 生命周期切片**：Core Animation backend、重墨样式、pause/rate、generation、stop/close fake-host 测试。
3. **真实纵向集成**：AppKit overlay host、Composition、现有播放器 surface，以及总开关和三类过滤的轻量 Feature 控制；不增加复杂设置中心。
4. **生产 probe 与收口**：从生产 renderer 调用路径提取快速合成入口；不得复制 spike harness。完成真实播放、resize/fullscreen、切换/关闭和 30 分钟 soak。

每个切片都必须独立编译、测试和接受未参与实现的新上下文只读审查；红区终审另外检查 MainActor、animation completion、owner、generation、resize 和 stop/close。

## 测试与 Gate

### 确定性测试

- scrolling 同类不追尾；top、scrolling、bottom 可以共享同一纵向位置，但各自的同类 lane 仍独立防碰撞；surface 太小、超长文字、空 lane 和突发 batch fail bounded。
- 同批时间/ID 顺序稳定；超过 640 的突发在 admission 前拒绝，allocation spy 证明被拒事件没有创建 `CATextLayer`，`peakActive <= 640`，无隐藏队列且只丢弃未显示事件。
- pause 冻结、0.5×/1×/2× 改变运动、前后 seek 与 identity 替换清空旧 generation。
- 同一 timeline generation 内 resize/clear 后的迟到 completion，以及 seek 后的旧 generation completion，都不能删除或修改新 layer；重复 stop/close 幂等，owner 与 layer 最终释放。
- resize/fullscreen 尺寸切换后旧 lane 状态不复用，0×0 surface 不 admission。
- 总开关关闭立即清空；三类过滤变化清空旧几何状态，不回填已错过事件；50%/75%/100% 的上下镜像 origin 由纯算法测试固定。

### 命令与现场证据

- 纯 package 变更：`sh Scripts/run-quality-gates.sh package`
- App/Composition/Package product 变更：`sh Scripts/run-quality-gates.sh app`
- 快速 probe：40/80 events/s，各少量重复，只记录 frame interval 辅助值、physical footprint、active/peak/drop 和清理。
- 快速合成入口复用生产 controller/backend：`Scripts/run-m44-renderer-probe.sh 40 30` 或 `Scripts/run-m44-renderer-probe.sh 80 30`；第二个参数是 1–1800 秒的运行时长。probe 只输出发射、接纳、丢弃、峰值和 stop 后清理计数，不输出正文或内容标识，也不承担真实播放、fullscreen 或内存判定。
- 真实播放：基础三类型、pause、倍速、前后 seek、resize、fullscreen、切换视频/分 P、关闭页面。
- 真实 Core Animation surface 还要观察 pause 前后 presentation position 连续、倍速切换没有跳变，并用少量 position 样本确认 0.5×/2× 位移斜率随速率改变；fake/计数测试不能替代。
- 30 分钟 soak：合成数据，不记录正文或内容标识；内存不得随已播放分段单调增长，stop/close 后 renderer 资源归零。

只有现场结论含糊、出现可见失败或需要定位 render/GPU 与 app update 时才运行 Instruments；不建立完整 Instruments 矩阵。

## 尚待技术审查决定

- scrolling 与固定弹幕默认展示时长、最小 lane 间距，以及最小 overlay 尺寸。
- 当前 Mac 之外的最低设备性能边界由后续真实设备证据决定，不从 spike 推导。
