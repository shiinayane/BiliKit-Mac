# M4.5 导航与窗口 Chrome 迁移实施草案

> Revision：AT1。状态：实施记录，不代表用户逐项确认；用户已于 2026-07-24 确认 A1，迁移已实现并通过完整 App Gate及独立技术／红区终审，正等待用户确认迁移后观感。

## 当前耦合清单

`ContentView` 当前约 283 行，拥有七个 ViewModel／导航状态、依赖组装、`NavigationSplitView`、侧栏、四个 route 分支、搜索提交与加载、认证恢复／失效、窗口关闭、历史未登录空态以及搜索／播放 toolbar。`WatchHistoryView` 另行拥有历史 toolbar 和页面激活任务。Scene 使用全局 `hiddenTitleBar`，因此页面无法独立表达标题显示规则。

真正需要拆开的不是“文件长度”，而是四种不同生命周期：

| 层 | 生命周期 | 允许拥有 |
| --- | --- | --- |
| Scene | 窗口集合 | 默认尺寸、系统窗口／toolbar 样式 |
| `ContentView` | 单窗口 UI 状态与模型生命周期 | 现有顶层任务、认证协调、跨页面／窗口清理 |
| 私有 App shell | 一级导航与当前 route | split view、侧栏、route switch、sheet 呈现绑定 |
| 私有页面根 | 当前 route 的 chrome | 语义标题、标题显隐、toolbar |
| Feature 内容 | 业务状态 | loading／empty／error／content、用户意图 |

## 分阶段迁移

### Phase 0：行为固定与 API sanity

- 用最小编译／真实窗口观察确认 macOS 15 可用的 `toolbar(removing: .title)` 与当前 `NavigationSplitView` 组合行为；验证语义标题仍能被窗口与辅助功能识别。
- 同时在默认／窄窗口确认历史刷新：默认窗口位于右侧操作区，窄窗口即使被系统收纳也仍可发现和使用。
- 若标题 API 无法稳定按 route 生效，或默认窗口的历史刷新不在预期操作区，停止并执行 A0 的回退决策，不继续拆文件。

### Phase 1：一个可回退的纵向迁移

- 新增一个私有 `AppShellView.swift`，从 `ContentView` 同时搬移两栏结构、sidebar、route switch 和热门／搜索／历史／播放四个轻量页面根。
- 热门／搜索声明各自语义标题并移除默认可见标题；搜索根拥有 `.principal` 搜索框。
- 历史根拥有“观看历史”可见标题和 `.secondaryAction` 刷新；`WatchHistoryViewModel.isBusy` 统一投影 `.loading`／`.loadingMore`，同文件 `HistoryRefreshButton` 直接绑定该值并在 busy 时禁用，不新增协议。`WatchHistoryView` 移除 toolbar，只保留内容、状态和 Feature 生命周期。
- 播放根拥有 `.navigation` 返回；`VideoDetailColumn` 不认识 App route。
- Scene 从全局 `hiddenTitleBar` 回到系统 title bar／统一 toolbar 样式，使可见标题由页面根控制。
- `ContentView` 原样保留当前 `.task(id: feedTaskID)`、认证 `.task`、两个 `onChange` 和 `onDisappear`；`AppNavigationModel` 与播放／字幕／弹幕实现不变。
- `playerContent` 仍只在 `ContentView.init` 由现有 factory 构造一次，并以稳定值传入 shell；不得在 `body`／shell 重新构造，也不得对播放子树添加 route／BVID `.id`。
- 迁移后逐项对照事件序列，确保卡片选择、返回、登出和窗口关闭仍只有原来的副作用点。

这一阶段只允许职责移动和三项 chrome 结果，不同时重排播放页或改变任务时机；整体作为一个提交回退。

### Phase 2：验证与删除旧 chrome

- 运行最小页面序列和 App Gate；只有 UI smoke 无法定位失败时，才补必要的 accessibility identifier。
- 删除迁移后无调用的旧 chrome helper；不改变任务、路由或播放 owner。

## 验证矩阵

首轮只跑一条页面序列：

```text
热门（无可见标题）
→ 搜索（无可见标题，输入框居中）
→ 历史（可见标题，右侧刷新）
→ 播放 A（左侧返回）
→ 返回来源
→ 再播放 B
→ 播放中关闭窗口
```

自动证据：

- App route 与历史 ViewModel 现有测试；
- toolbar／标题最小 UI smoke，并固定历史 busy 时刷新禁用；
- 在现有 `PlayerHostLifecycleProbe` 上只追加累计 create/dismantle 事件；通过一条完全离线、托管真实 `ContentView` 的 App route 序列，依次得到 create A、dismantle A、create B、dismantle B；最终从同一 `NSHostingView` 移除 `ContentView`，触发生产 `onDisappear` 关窗清理，并联合检查 route、stop 次数和 ViewModel 归零。注入只服务 App test，不进入 `AppEnvironment.live()`、生产日志或通用诊断入口；
- 同时保留 player host `active == 1`、整条序列 `peak == 1`、关闭后 `active == 0`；
- `sh Scripts/run-quality-gates.sh app`。

真实证据只检查默认窗口和一个较窄窗口，不扩展浅／深色、全屏和辅助显示完整矩阵；那些属于解冻后的视觉精修。

## 审查与回退

- 价值／简化 reviewer：删除没有改变 owner 或页面规则的抽象。
- 可理解性 reviewer：确认用户只需理解四层职责、三个页面结果和停止条件。
- 技术 reviewer：检查 SwiftUI modifier 传播、Feature 依赖、任务与状态 owner。
- red reviewer：检查 View 提取是否改变 player／subtitle／danmaku、认证和窗口清理。

Phase 1 是唯一生产结构提交并可整体回退。若出现双 toolbar、旧页面 item 残留、重复任务、双播放副作用或 host identity 改变，只能在独立分支 revert 该唯一迁移提交，或直接放弃该分支；不得 hard reset 工作树，契约、验证记录和用户其他改动必须保留。不在故障结构上继续补条件。
