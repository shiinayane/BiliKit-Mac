# M4.5 Slice B 播放页响应式重排决策

> Revision：B3。状态：用户已于 2026-07-24 明确同意 B1 契约，并在快速视觉调整中最终确认主区水平边距 40 pt、垂直边距 24 pt、右栏 400 pt，以及标题以下页面文字提升一个语义字号。16:9 播放器、单分 P 无右栏与生命周期语义不变。Slice A 已于提交 `22bfa24` 关闭；Slice B 可以按本文件进入生产实现，不沿用 Slice A 的实现授权。技术草案见 [`M4.5-playback-layout-technical-plan.md`](./M4.5-playback-layout-technical-plan.md)。

## 为什么现在做

Slice A 已把搜索、热门、观看历史和播放入口收敛到唯一 App shell，并固定单一播放器 host 与返回 owner。当前播放 route 仍沿用 M4 的验证布局：播放器在最上方，标题、简介、字幕、弹幕和分 P 全部纵向堆叠。它能验证能力，但没有利用宽窗口，也会在单分 P 视频中显示没有决策价值的模块。

本轮只重排已经存在的信息。它不改变播放、字幕、弹幕、路由或数据能力。

## 本轮只支持两个决策

1. **响应式结构**：宽窗口是否使用“主观看区 + 只读分 P 右栏”，窄窗口是否把同一分 P 列表收进主区的原生可展开区域。
2. **生命周期保护**：怎样保证宽窄与全屏变化只改变布局，不复制或替换播放器 host，不重载播放项，也不重启字幕／弹幕会话。

本轮不比较多套视觉主题，不决定可选择分 P、独立播放器窗口、mini player、PiP、评论、相关推荐或 UP 主资料。

## 候选与选择

### 方案 A：继续单列，只隐藏单分 P

它也可以让所有多分 P 都使用主区内 disclosure，从而只维护一棵纵向页面。改动最小，但宽窗口仍浪费横向空间，也不能完成已确认的播放页信息层级。淘汰；“始终 compact”只在响应式宽度无法可靠判断时作为需要重新确认的回退候选，不是本轮生产范围。

### 方案 B：一个稳定主区，按可用宽度增减分 P 辅助区（选择）

播放器、标题、元信息、字幕／弹幕控制和简介始终属于同一个带页面边距的主区，`playerContent` 在 View 结构中只声明一次。播放器外层是稳定的 16:9 黑色观看容器，按主区可用宽度等比缩放，不以任意高度拉伸。多分 P 时：

- 可用内容宽度足够时，在主区右侧显示固定上限宽度的只读分 P 列表；
- 可用内容宽度不足时，右栏消失，同一列表改在主区控制项之后以原生可展开区域呈现；
- 单分 P 时不显示标题、空白占位或可展开入口。

宽窄切换只增减播放器旁边的辅助 sibling，不能在两个响应式分支中各自创建播放器。

### 方案 C：宽版与窄版各维护完整播放页

实现直观，但两棵树会各自持有播放器内容，切换时可能 create/dismantle host，并使信息顺序逐渐分叉。淘汰。

### 方案 D：建立通用响应式布局框架

当前只有一个播放页需要该规则；自定义 Layout、环境 registry 或通用断点系统不能增加本轮证据。淘汰。

## 获得授权后实施的范围

- 主区信息顺序固定为：标题、UP 主／播放量／弹幕数、播放器、字幕控制、弹幕控制、简介。
- 播放器观看容器固定为 16:9；非 16:9 视频由现有 `AVPlayerView` 在黑色容器内 letterbox／pillarbox，不裁切画面，也不改变媒体或 renderer。
- 主区使用局部 40 pt 水平边距与 24 pt 垂直边距，播放器与标题、控制和简介共享该阅读边界；左侧必须露出窗口 detail 背景，不能让黑色播放器贴住 `NavigationSplitView` 分隔线而与 sidebar 背景融为一体。
- 标题保持 `title` 层级；元信息提升为 `body`，简介、分 P 与字幕／弹幕控制提升为 `title3`。播放器画面内字幕保持独立媒体可读性设置，不跟随页面字号。
- 右栏只消费现有 `pages` 和 `selectedPage`，只读显示当前分 P、标题和时长；不增加点击、键盘选择或 CID 切换。
- 窄窗口的分 P disclosure 默认保持可发现；展开状态只属于 View 生命周期，不持久化、不进入 ViewModel。
- 响应式判断只读取扣除主区页面边距前的播放页实际可用宽度。1080×680 基线必须进入 compact disclosure，1320×820 基线必须进入 wide rail；首轮只使用一个局部阈值、一组水平／垂直内容边距和一个 400 pt 右栏宽度，不建立全局 token 或断点表。
- 复用 Slice A 的 `PlayerHostLifecycleProbe`，只补充同一播放会话跨宽窄时 identity 稳定的断言，不增加第二套计数器。

## 必须保持的不变量

- `AppNavigationModel` 继续是唯一一级 route、开始播放和停止播放 owner；返回入口与来源快照不变。
- `ContentView` 继续在窗口生命周期内只构造一个 `playerContent`；Slice B 不调用 player factory，不增加 `.id`，不移动 engine、renderer、subtitle、danmaku 或清理 owner。
- 同一次播放会话中，宽窄变化不得改变 player host identity、触发新的 playback load，或重新选择字幕／弹幕 identity。AVKit 原生全屏必须保持用户可见的播放连续、不重载；当前自动化不把全屏内部 host identity 声称为已证明。
- 分 P 显隐不得调用 `selectVideo`、reload、seek、切 CID，或重建字幕／弹幕 task。返回后进入另一视频允许产生新的播放 identity。
- `pages.count == 1` 时完全隐藏分 P 模块；只有 `pages.count > 1` 才能进入 compact／wide 分 P 布局。空 pages 不是 `singlePart`，继续由既有无效响应路径失败关闭，不进入布局层。
- Feature 之间不新增依赖；不新增 target、Package、endpoint、Application port、持久化或第三方 UI 依赖。

## 复杂度预算

- 最多修改播放详情的两个现有 Feature 文件、一个现有 App 生命周期测试文件，以及必要的 App／Feature 测试；不改 Xcode target 或 Package product。
- 最多新增两个文件内私有布局类型：一个响应式 mode／metrics 值和一个只读分 P 视图；若现有文件职责仍清晰，可不新增生产文件。
- 只保留一个宽窄阈值、一个右栏宽度范围和一个关键迁移序列；不建立窗口矩阵 runner、截图框架、性能 harness 或新 probe。
- 自动化只扩展已有 host hook 与全假值 View 测试；M4 长时 probe 只在出现资源异常或触碰 engine／renderer 时重跑。

## 最低证据、停止与回退

完成至少需要：

- 单分 P 在宽窄窗口都不显示分 P 模块；多分 P 在宽窗口显示右栏、窄窗口显示可展开区域，当前项标识和时长正确。
- 1080×680 与 1320×820 的无个人数据 View／签名 App 检查中，播放器外层保持 16:9，主区左右保留 40 pt、上下保留 24 pt，播放器左右边缘不贴住相邻 surface；没有播放器挤压、任意比例拉伸、内容截断或不可达控制。
- 签名 App 至少观察一个非 16:9 视频，确认完整画面以 letterbox／pillarbox 显示而不是 aspect-fill 裁切；不记录内容标识或画面。
- 多分 P 的播放 A 在布局 marker 确实经历 compact → wide → compact 后，同一会话 host identity、playback load identity、字幕／弹幕选择次数不变，active 与 peak 都为 1。返回必须 dismantle A；播放 B 可以 create 新 host，但每个会话的 create／dismantle identity 必须配对，整条序列 peak 为 1，关闭后 active 为 0。A→B 的旧结果隔离继续由既有回归测试覆盖，不扩张成 Slice B 新矩阵。
- 签名 App 的“播放 A → 调整宽窄 → 进入／退出全屏 → 返回 → 播放 B → 关闭窗口”只证明用户可见的播放连续、来源与清理；若无法用现有脱敏 hook 观察全屏内部 identity，则把它保留为未验证边界，不以人工观察冒充对象证明。
- 完整 App Gate、实现后独立 `red_reviewer` 和用户真实关键路径确认通过。

若必须复制播放器树、改变 route／清理 owner、加入分 P 选择能力、触碰 engine／renderer，或无法以现有 hook 证明单 host，立即停止。回退时完整恢复 Slice A 的旧详情布局，不保留半套右栏或第二响应式实现。

## 三项授权

1. 是否授权“带 40 pt 水平／24 pt 垂直页面边距的 16:9 播放器、400 pt 宽屏只读分 P 右栏、窄屏可展开分 P、单分 P 完全隐藏，以及标题以下页面文字提升一个语义字号”作为 Slice B 的唯一产品范围？
2. 是否同意 Slice B 只重排 View，并保持 route、单一 player host、播放 item、字幕／弹幕 owner 与数据能力全部不变，即缩放或全屏时当前播放不中断、不重载？
3. 是否同意按上述单一关键路径验证；一旦需要双播放器树、分 P 选择、通用布局框架或新的播放机制就停止并重新确认？
