# M4.4 renderer spike 决策契约

> Revision：D3。状态：用户已于 2026-07-23 在 Codex task `019f89a8-67ed-73e1-9915-5d7ca36ccd1e` 的连续入站回复中明确同意，并在解释后纠正了唯一影响路线选择的重大误解；治理热修合入后允许新建独立 worktree 开始 Phase 0 A，以及仅在既定 Gate 触发时实施条件性 Phase 1 B。不授权 Phase 2、恢复旧 harness 或开始生产 renderer。改变目标、候选、淘汰/停止条件、影响区域或下一步时必须递增 revision 并重新确认。

## 通俗摘要

先只试最简单的系统文字层：它若能在 40 events/s 目标负载和 80 events/s 余量负载下流畅显示，正确暂停、跳转和关闭，并能清干净资源，就直接停止实验。只有流畅度或内存失败在两档都重复出现、且可能由文字画法改善时，才试“先把文字画成图片”的第二条路线；语义或清理失败直接停止。实验只在独立、不可合入目录中使用假数据，不碰生产 App、网络或真实账号。结果只用于选择路线，不承诺所有 Mac 的容量。

## 为什么做、决定什么

M4.3 已经解决弹幕数据和时间轴。M4.4 只需要避免过早选中一个明显卡顿、无法正确跟随播放或资源不能回收的文字承载方式。

本 spike 只回答一个路线选择，并分两步停止：

1. 最简单的系统文字层（`CATextLayer`）若通过本轮固定的目标与余量负载，就进入生产设计并停止研究。
2. 只有第一步在两档各两次复现可能由栅格化改善的流畅度或内存失败时，才判断预绘制文字图片并放入普通 layer（Core Text bitmap-backed `CALayer`）能否解决该失败；语义/清理失败或 B 未完整通过两档时停止并重新讨论候选。

它不决定跨设备产品上限、最终视觉样式、完整 lane/碰撞策略或生产 Gate。

## 本轮固定输入

- 20 events/s 只作显示 smoke，40 events/s 是临时目标负载，80 events/s 是余量负载。8 秒可见时长下约对应 160、320、640 个活动对象；这些数字只定义当前 Mac 上的实验，不是产品承诺。
- 最低可读样式固定为 24 pt 系统 semibold、白色文字、透明背景，以及两候选共用的 1 physical-pixel 黑色 layer shadow；假文本覆盖中文、日文、韩文、拉丁字符和 emoji。这不是最终视觉验收。

选择依据是：目标档足以暴露数百个同时活动对象的数量级，余量档再翻倍，同时 bitmap backing 的推导峰值仍低于 T2 的 128 MiB 安全停止线。

## 最低证据与淘汰条件

在一个主要播放尺寸中，只看三类结果：流畅度、内存、资源数量与清理。还必须验证 pause 会冻结，seek 会清掉旧内容，seek/close 前的旧工作之后不能重新出现，stop/close 会释放 surface 和任务。

目标或余量负载下重复出现明显卡顿、资源超过由固定输入推导的上限、旧内容回流或清理失败时淘汰候选。结果仍落在正常重复波动内时不强行选择，返回决策价值 Gate。

## 允许影响的区域

获准后，只能在独立不可合入 worktree 的 spike-only 目录和一个 runner 脚本中创建合成 host、候选与结果。不得修改 `Package.swift`、生产 App/Composition/Platform、现有 Package source/API、网络、认证、持久化或真实数据；如确需越界，先停止并重新确认。

## 停止条件与复杂度预算

- 一次只实现一个候选；第二个候选仅在第一个失败后启用，总数最多两个。
- 最多三个核心指标、三个负载档和一个 surface。
- Phase 0 最多一个进程、一个 host/入口、一个固定输入 fixture、一个简单 recorder 和一个 runner；不开发 IPC、schema、broker、进程监管或通用 benchmark framework。
- 新增 executable/process、协议、结果 schema、surface 或另一项实验维度即视为超预算，必须说明它保护哪个决策并重新请求用户确认。
- 用户决策契约保持约两页；实施细节放入技术计划。

候选通过目标和余量负载就停止；两条都失败、结果不能改变行动、或 harness 接近候选代码复杂度时也停止。结果出来后另起 M4.4 生产契约，用户再次确认后才写可合入 renderer。

## 可理解性 Gate

上方“通俗摘要”不超过 300 字。用户明确同意并且没有影响授权或路线选择的重大误解，即视为确认 D3；遗漏不主动判错，技术实施细节无需掌握。授权问题最多三个：

1. 是否同意先试系统直接画字，仅在重复出现、且可能由预绘制成图片改善的流畅度或内存失败时才试第二条路线，其他严重失败直接停止？
2. 是否同意本轮范围、主要风险、淘汰/停止条件和 spike-only 隔离边界？
3. 是否同意本轮只授权 Phase 0 A 与满足既定 Gate 后的条件性 Phase 1 B；Phase 2 和生产 renderer 仍需另行确认？

用户未主动复述某一点不构成失败；只有明确的重大误解需要解释并重新确认该点。不得要求背诵 D3/T3、完整枚举条件或重复已经确认的答案。
