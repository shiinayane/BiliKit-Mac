# M4.5 UI/UX 生产实施草案

> Revision：PT7。状态：实施草案，不代表用户逐项确认；P6 三项授权已由用户于 2026-07-23 确认。Slice A 已冻结于提交 `0b1f94a`；A1 页面／导航／窗口 chrome 迁移已经实现并通过本地 Gate，正等待用户确认迁移后观感。确认前 Slice B/C 不得启动。架构与迁移记录见 [`M4.5-navigation-chrome-architecture-decision.md`](./M4.5-navigation-chrome-architecture-decision.md) 与 [`M4.5-navigation-chrome-migration-plan.md`](./M4.5-navigation-chrome-migration-plan.md)。

## 当前事实与目标边界

- `ContentView` 组装所有 ViewModel，并以唯一 `AppNavigationModel` 持有搜索、热门、历史和 playback route；认证保留 sheet，历史已是主页面。
- 根界面已是两栏 `NavigationSplitView`；搜索置顶，热门和历史位于其下，账户入口固定在侧栏底部。
- `PopularFeedView` 与 `VideoSearchView` 复用 16:9 卡片流；搜索词与来源由内存返回快照恢复，卡片／历史选择只发送 App route intent。
- `AppEnvironment` 在窗口生命周期内只创建一个 `AVPlayerEngine`、renderer、controller 与 `PlayerHostView`。
- 当前 playback route 仍沿用 M4 详情布局；响应式信息重排、单分 P 隐藏与多分 P 右栏属于 Slice B。
- M4 已固定 resize、fullscreen、seek、替换、时间轴和资源清理基线。M4.5 不改变 engine、renderer、endpoint 或 Application 用例。

## 所有权与最小接口

App target 新增唯一仅限 App module 内部、非公开的 route，使 App tests 可通过 `@testable import` 固定转换；语义限定为搜索、热门、历史和播放，具体命名可在实现时调整：

```text
AppRoute
├── search
├── popular
├── history
└── playback(bvid)
```

route 只保存当前目标；唯一内存返回快照独占来源 route、搜索 query 和选中 BVID，避免两份 source 分叉。优先依赖系统容器保持滚动；只有真实返回会失去方向、且已有稳定 identity 能直接作为 anchor 时才增加最小滚动恢复。App shell 收到卡片／历史选择 intent 后进入 playback；返回恢复来源。登出清除历史来源快照，关闭窗口清除全部快照。

Feature 继续拥有自己的 ViewModel、加载状态、取消和错误文案。为让 App shell 组合页面，只开放当前真实调用方需要的最窄 Scene／intent API；不公开具体 Repository、播放器或跨 Feature route。若需要让 BrowseFeature 认识 History，方案作废。

## 分阶段实现

### Slice A：完整可工作的两栏路由迁移

- 原子建立唯一 App route、两栏 `NavigationSplitView`、搜索／热门／历史／账户、卡片选择、沿用现有详情布局的 playback route 和返回；删除旧三栏导航，不产生双 route owner 或不可播放的中间状态。
- 卡片和历史选择只发送 route intent；只有 route transition 调用 `selectVideo`。返回只由 App shell 的一个出口 reset video／subtitle／danmaku，禁止保留现有两层 `onChange` 转发。
- 在现有 `NSViewRepresentable.makeNSView`／dismantle 边界增加一处 App 内部、每窗口独立且仅测试使用的生命周期 hook：播放期间 `active == 1`，A→返回→B 整条序列 `peak == 1`，返回／关闭后 `active == 0`；不增加第二套计数器，不做进程全局统计，也不记录 BVID、标题或播放内容。
- 认证保留 sheet。未登录选择历史时由 App shell 显示内联引导，用户主动点击才打开认证 sheet；登录后才挂载 `WatchHistoryView` 和发起请求。历史改为主页面，并把 `onDisappear` 无条件 reset 调整为符合 route 生命周期的显式加载／取消；登出仍立即清除个性化数据。
- History ViewModel 增加最小 route-deactivation 语义：取消初始 loading 后回到 idle，取消 loading-more 后回到之前的 loaded 内容；登出、认证失效和关闭窗口仍 reset。只补这三类负向转换，不新增 protocol 或 coordinator。
- 热门与搜索使用同一 owner 内的最小 `VideoCard`：`LazyVGrid`、16:9 封面、两行标题、一行元信息、时长角标、稳定占位。
- 搜索输入使用系统 searchable／toolbar 语义；清空后回到 prompt，不触发空 query 请求。
- 固定 route、A→返回→B 的唯一播放副作用、host active／peak、旧结果隔离、历史离开／返回和认证失效测试；测试命名明确 identity 只在同一次播放会话的布局变化期间稳定，不建立滚动 anchor owner。

### Slice B：播放页响应式重排

- 在 Slice A 已经可播放、可返回的独立 route 内重排页面；不再改变 App route 或返回 owner。
- 播放页只声明一次 `playerContent()`。响应式布局保持同一主观看区节点，只调整右栏宽度、可见性或 disclosure；不得用两个条件分支各创建播放器。
- 复用 Slice A 的同一 hook，追加断言同一次播放会话在宽窄／全屏期间 host identity 不变；返回后进入另一视频允许新 identity。
- 主区重排为标题／元信息、播放器、字幕与弹幕控制、简介；右栏仅在 `pages.count > 1` 时显示现有只读分 P 列表与当前项，不新增选择 action。
- 分 P 显隐不得切换 CID、reload item 或重建 subtitle／danmaku session。零分 P 继续走既有无效响应路径。
- 用“播放中跨宽窄／全屏后返回，再选择另一视频并关闭窗口”一条关键序列断言 host peak 为 1、布局期间 identity 不变、退出 active 为 0，并覆盖旧结果和清理；不把 M4 已验证行为展开为新笛卡尔积。

### Slice C：精修与 Gate

- 统一真实 loading、empty、failure、retry、signed-out、hover、pressed、selection 与 focus；不新增 offline 等没有现有状态来源的伪状态。
- 最小 token 只覆盖已复用的页面边距、卡片间距、圆角、封面比例、阅读宽度和弱化文本层级。
- 完成浅／深色、Increase Contrast、Reduce Transparency、Reduce Motion、文字放大，以及主要控件 VoiceOver 标签与键盘激活。
- 用自制 Preview 假值和无个人数据 XCUI 路径验证，不把真实账号、标题、BVID、字幕或弹幕正文写入 fixture、截图文件名或日志。

## 关键状态转换

| 事件 | route／页面动作 | 必须保持 |
| --- | --- | --- |
| 热门／搜索／历史选择视频 | 只改变 App route；route transition 开始播放 | 只有一个播放副作用点 |
| playback 返回 | stop/reset 当前详情，恢复来源 | query、选中卡片可恢复；精确滚动仅条件启用 |
| 快速 A→返回→B | 取消 A，B 成为唯一 owner | A 的结果不能覆盖 B |
| 历史加载中离开／返回 | route deactivate 后恢复可加载状态 | 不停在 loading／loading-more |
| 登出／认证失效 | 离开个性化历史，清快照与历史内容 | 显示内联引导，不自动弹 sheet |
| 关闭窗口 | 取消页面 Task，停止播放 | observer、loopback、surface 归零 |
| 宽窄／全屏变化 | 只更新布局 | host／engine/item identity 不变 |

## 验证与审查

- 每个代码 slice 至少运行 `sh Scripts/run-quality-gates.sh app`；先写或定位能固定 route、取消、旧结果和单 host 的失败测试。
- Slice A 用 Preview、ViewModel／route 测试、host hook 和 XCUI smoke 验证 1080×680、1320×820、搜索／热门／历史／账户、播放与返回，确保合入时不存在双导航、死路或 host 生命周期回归。
- Slice B 复用 M4 真实 App 路径，只增加“跨宽窄／全屏→返回→选择另一视频→关闭”关键迁移序列，以及 host active／peak／identity 的单一测试 hook。
- Slice C 做成对视觉与辅助功能检查；不建立截图服务、manifest、像素 diff 或通用 benchmark。
- M4 的 30 分钟结果只作回归基线；只有触碰 engine／renderer，或关键迁移序列出现资源趋势异常时，才重跑既有 probe，不新增 soak。
- Slice A 与 Slice B 都由独立 red reviewer 检查线程、owner、播放生命周期和清理；Slice C 由独立 reviewer 做黄区审查。最终再做价值删减与可理解性复核，避免为“完善”扩大范围。
- 自动化和 Agent 真实验证通过后，由用户沿同一关键序列确认播放连续、来源与导航正确，再关闭 M4.5。

## 回退

- Slice A 失败：整体回退 App route／卡片迁移，恢复当前三栏闭环；不保留双导航或无播放中间态，不改数据层。
- Slice B 无法稳定单 host：完整恢复 Slice A 已由 hook 验证的旧详情布局；不尝试另一套未经验证的新布局，不改 engine 或 renderer。
- 任一 slice 需要 endpoint、持久化、新 target 或第二套导航时停止，回到 P6 重新确认。
