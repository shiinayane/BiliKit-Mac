# 工程质量 Gate

本页约束 BiliKit 的开发过程，目标是让高风险改动保持可理解、可验证和可回滚。它不要求为每个函数写说明，也不以测试数量、覆盖率或代码行数代替判断。

## 一次变更只解决一个目的

开始前用 PR 模板写清：

- **Goal**：这次要改变什么可观察行为。
- **Context**：相关入口、ADR、威胁模型、历史验证和已知限制。
- **Constraints**：依赖方向、安全/隐私、取消、资源和兼容性边界。
- **Done when**：哪些自动测试、真实探针或测量能够证明完成。

如果无法用一句话说明目的，或同时改变两个可以独立验证的行为，先拆分。代码行数和文件数量只触发职责检查；生成代码、fixture 和机械迁移不按同一尺度判断。

## 决策价值 Gate 与复杂度预算

黄区中的未知路线、红区和重要 Gate 在详细技术设计前，先回答：

1. 本任务最终只支持哪一两个具体决策？
2. 哪些结果会改变下一步，哪些指标即使测出也不会改变决策？
3. 为什么是这些候选；是否遗漏同成本等级、可能改变结论的路线？
4. 能否先用更便宜、更短的观察淘汰明显不合适的候选？
5. 实施或 harness 成本是否接近、超过被验证的生产切片？
6. 哪些事实尚无产品依据，不能伪装成精确阈值或产品承诺？

无法回答时停止，不以“更加严谨”或“reviewer 可能质疑”为理由进入详细设计。

任务契约同时写明复杂度预算，至少限制主要候选、核心指标、运行矩阵和实施组件。预算必须列出允许创建的 executable/process、runner、fixture、结果格式和额外协议，不能只写“保持简单”。未知性能路线默认最多两个主要候选、三个核心指标、三个主要负载档；Phase 0 默认单进程、一个 host/入口、一个固定输入 fixture、一个简单 recorder，不开发 IPC、schema、进程监管或通用 benchmark framework。若必须超出预算，先说明新增成本保护哪个决策、为何没有更便宜控制，并重新请求用户确认。

不确定性使用逐级加码漏斗：Phase 0 只做 sanity 与明显淘汰；Phase 1 做最小可重复比较；Phase 2 只解决结果接近、重复矛盾、工具干扰、偶发失败或难回滚架构所需的真实不确定性。不得在 Phase 0 前预建 Phase 2 的完整基础设施。

## 风险分级

### 绿区

文案、Preview、局部布局、仅限私有实现的机械重命名、明确的 DTO 映射或小型测试补充。主实现者可以直接完成并检查 diff；只要触及代码，仍须运行根规则规定的 Package 基线。公开 API、target 或工程变更不能因改动机械而归入绿区。

### 黄区

普通 Feature、Use Case、缓存策略、跨文件重构、新增跨模块公共 API。先写任务契约；实现后必须由不继承实现推理的新上下文做只读审查，再修复并运行完整 Package 或 App Gate。

### 红区

认证、授权、Keychain、远端来源策略、重定向、本地服务器、播放/媒体、线程与资源生命周期、弹幕 renderer、持久化迁移、文件删除和其他不可逆数据变化。无论改动多少行，都必须：

1. 通过决策价值 Gate 并写明复杂度预算。
2. 由独立上下文做价值与删减审查；不得把所有建议直接累加进契约。
3. 由另一独立上下文检查决策摘要、授权范围和主要风险是否可理解；技术实施细节不进入用户测试。
4. 由独立 `red_reviewer` 对待确认的最小契约做技术风险审查。预确认阶段不要求已有用户确认或实验结果；若需要显著扩张范围，返回第 1 步。
5. 整改若改变目标、候选、淘汰/停止条件、影响区域或下一步，重新执行受影响的前置审查。之后向用户提供不超过 300 字的决策摘要和最多三个授权问题；用户明确同意且没有影响决策的重大误解即通过。Agent 不得代替确认。
6. 必要时在独立 `codex/spike-*` 分支做不可合入实验；实验结束后再由用户确认生产契约。
7. 由实现者按生产契约完成最小纵向切片，并由独立上下文审查失败、取消、乱序、线程、所有权、安全、资源上限和清理路径。
8. 运行自动化 Gate、任务专用真实探针或确有决策价值的长时测量。
9. 由用户沿一条最关键的真实链路确认需求与风险边界。

同一模型的新上下文只能降低相关误判，不能替代真实环境、不同验证方法或用户判断。

### 可理解性与知情授权

- 用户只需理解并确认决策层：目标、候选、范围、主要风险、淘汰/停止条件、实验隔离和下一步。
- seed、采样公式、对象池数字、脚本参数、内部清理顺序等技术细节默认由实施计划和技术审查负责；只有它们改变授权范围、成本、安全或最终决策时，才需要用户再次确认。
- 用户没有主动复述某一点不构成失败。只有用户明确表达了会改变授权或决策的重大误解时才暂停；解释后只重新确认该点。
- 次要遗漏只提示并允许确认；技术实施细节无需用户掌握。不得要求用户记忆原文措辞、完整枚举条件或重复已经确认的内容。
- 确认记录指向一条或多条实际入站回复，并绑定决策契约 revision。技术计划发生语义变化时必须递增 revision 并重新技术审查；仅当变化影响范围、成本、安全或最终决策时，用户确认才失效。
- 可理解性 Gate 的目标是知情授权，不是闭卷测试、背诵或表单劳动。

## Agent 与工作树隔离

- 独立审查使用不继承主对话的上下文；只提供 Goal、Context、Constraints、Done when 和待审文件，不提供实现者结论。
- 测试审查以 Roadmap、ADR、威胁模型和外部行为为事实源，不从实现反推期望。
- 只读审查可以共享工作树；多个写入 Agent 禁止同时修改同一工作树。
- 确需并行写入时，每个 Agent 使用独立 branch/worktree；一次只审查和合入一个重要变更。
- 审查输出必须按 `blocker / improvement / reject` 分类。没有可执行问题时明确写“未发现阻断”，不能只写“看起来没问题”。
- 三种审查视角使用互不继承结论的独立上下文，并按价值与删减 → 可理解性 → 技术正确性的顺序执行。后一个 reviewer 只接收当前契约和事实来源，不接收前一个 reviewer 的结论；视角是任务 mandate，不要求新增一套自定义 Agent 配置。
- 价值与简化视角找不改变决策的条款、伪精确阈值、错误候选和成本高于风险的防护；目标是获得有用结论的最短路径。
- 可理解性视角只检查决策层能否支持知情授权，区分重大误解、次要遗漏和技术细节；不负责技术终审，也无权代替用户确认。
- `reviewer`/`red_reviewer` 不能只因理论完备性增加机制。任何新增项必须说明具体失败、错误决策路径、更便宜替代和必须现在解决的理由。
- reviewer 结论冲突或共同导致预算超限时，主 Agent 返回决策价值 Gate；不得把三份意见机械合并成更大的方案。

## 模型路由与升级

项目配置位于 `.codex/config.toml` 与 `.codex/agents/`。默认主 Agent 为 Sol Medium；子 Agent 按任务的认知难度路由：

| 角色 | 默认配置 | 适用范围 | 不得独立承担 |
| --- | --- | --- | --- |
| `explorer` | Luna Low，只读 | 找符号、入口、依赖、有限调用链和整理日志 | 架构取舍、竞态、安全或生命周期结论 |
| `worker` | Terra Medium | 按已确认契约实现可独立验证的窄切片 | 改变公共 API、架构、安全边界或验收条件 |
| `reviewer` | Terra High，只读 | 黄区的契约、失败、取消、测试与维护性审查 | 红区最终签字 |
| `red_reviewer` | Sol High，只读 | 线程、生命周期、认证、安全、播放、renderer、迁移和重要 Gate 终审 | 代替真实探针、长时测量或用户确认 |

满足任一条件时停止当前低成本路线并升级主 Agent 或 `red_reviewer`：

- 需求或证据存在两种以上合理解释。
- 无法明确写出状态不变量、资源 owner、清理点或 rollback。
- 变更跨越两个以上 target 边界，或触及红区。
- 自动测试失败原因不清晰，或同一路线连续两次修复失败。
- reviewer 之间有影响正确性的冲突，或准备关闭重要 Roadmap Gate。

便宜模型只处理“已经知道该找什么、怎样验收”的工作。绿区不为了形式完整而启动全部角色；subagent 默认限制输出范围，避免重复扫描全仓库。模型名、推理档位或基准分数都不是质量证据，实际任务中的 blocker 命中率、误报、升级、返工和 Gate 结果才是后续调整依据。

## 可执行 Gate

统一入口：

```sh
sh Scripts/run-quality-gates.sh static   # 文档、脚本和纯静态改动
sh Scripts/run-quality-gates.sh package  # Package 代码；包含 static
sh Scripts/run-quality-gates.sh app      # App/Xcode/composition；包含 package
```

风险等级只会追加验证，不会降低基线：所有代码改动至少运行 `package`，涉及 App/Xcode/composition 时运行 `app`。脚本尊重当前 `xcode-select`；只有调用方显式设置时才使用 `DEVELOPER_DIR`。本地 whitespace 检查覆盖 tracked 的 unstaged/staged diff，untracked 文件在暂存后或 CI 的提交范围检查中覆盖。上述命令只证明构建、确定性测试和静态契约。红区任务必须在任务契约中追加具体命令和阈值，例如受控播放探针、弹幕高密度基准、30 分钟 soak、签名 Keychain smoke 或真实 UI 检查。普通 `swift test` 输出保留在 CI；只有难以自动化、以后需要复查的真实设备、网络、性能、安全或兼容性证据才写入 `docs/validation/`。

任何删除、跳过、放宽或绕过现有测试、架构、秘密、工程契约或真实 Gate 的变更均为 blocker，除非同一变更解释原因并提供等价或更强的替代证据。

## 分支、PR 与提交

- 用户授权 Git 交付时，黄区和红区从短分支开始，通过 PR 合并；不在 `main` 上持续堆叠开发。没有提交、推送或 PR 授权时，只在本地保留改动和审查结果。
- PR 描述使用仓库模板，列出风险等级、关键路径、验证结果和未覆盖边界。
- 提交按可独立解释和回滚的行为拆分；生成物与其 schema/runtime 变更放在同一提交。
- `CHANGELOG.md` 只在形成可发布版本后记录用户可见变化；内部函数历史以 Git、测试、ADR 和验证记录为准。
