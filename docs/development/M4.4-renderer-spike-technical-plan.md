# M4.4 renderer spike 技术实施计划

> Revision：T3。状态：独立技术审查无 blocker；用户已确认 D3 对 Phase 0 A 与满足既定 Gate 后的条件性 Phase 1 B 授权，不要求逐项掌握 T3。治理热修合入后只允许在新独立 worktree 实施这两步，不授权 Phase 2。命名、输出排版等非语义细节可调整；技术变化必须递增 revision 并重新做定向红区审查，只有影响授权范围、成本、安全或最终决策时才使用户确认失效。候选、淘汰/停止条件、允许影响区域或下一步变化还必须更新决策契约。

## 固定边界

- 只在 spike-only 目录和一个 runner 中工作，不修改生产 source、`Package.swift`、App 集成或真实数据路径。
- 一个普通可见窗口、1280×720 内容区、当前屏幕 scale、一个 Release 单进程 host。
- 固定 fixture 按“简体与繁體中文 / 日本語かなカナ漢字 / 한국어 테스트 / BiliKit Renderer / 😀🎬✨”五个模板等比例循环，每条追加唯一序号；单条测量 bounds 加统一 2 physical-pixel padding 后不得超过 640×64 physical pixels。两候选共享同一 `NSAttributedString`、系统字体 fallback、Core Text shaping/测量结果、baseline、padding 与 `contentsScale`。最低样式为 24 pt 系统 semibold、sRGB 白色 fill、透明背景，并在承载 layer 上使用相同的黑色 shadow：opacity 1、零 offset、`shadowRadius = 1 / contentsScale` points。两候选都不做跨事件文字缓存；B 只改变单条文字如何同步栅格化并放入 layer。
- 各速率使用确定性等间隔时间戳；活动区间为半开区间 `[start, start + 8s)`，同一时间先移除到期对象再 admission。20/40/80 events/s 的活动峰值分别为 160/320/640；允许一个 admission 临时量时总 layer cap 分别为 161/321/641。
- 合成滚动对象可见 8 秒，每轮预热 10 秒、观察 60 秒；这些只定义同机实验负载，不是产品动画或长时稳定性承诺。
- 正式证据与用户契约一致，只有三类：流畅度（frame interval 与人工运动 smoke）、内存（physical footprint）、资源数量与清理（候选实际拥有的计数）。
- renderer owner、surface、计数器和 animation completion 都由 `@MainActor` 隔离；异步 animation 回调先回到该隔离域，再核对 generation。B 也在 MainActor admission 内同步完成 Core Text 栅格化，不创建 bitmap Task、队列或异步 completion；临时 context 在赋值后立即释放。

## Phase 0：先证明最简单路线

只实现 `CATextLayer`。20 events/s 只做一次 15 秒显示 smoke，不产生路线结论；40 events/s 目标负载和 80 events/s 余量负载各运行两次：

1. 显示最低可读样式；
2. pause、seek 清空/重建、stop/close；
3. 挂起旧 animation completion，seek 并保持 owner 活跃，释放 completion 后确认它不能向新 generation 提交；随后分别 stop、close，确认实际拥有的对象/任务归零且 surface/owner 弱引用释放；
4. 记录三个核心指标，不采集额外“辅助指标”。

若语义、清理、目标与余量负载均无明显问题，选择该路线并停止 spike。语义、generation、owner 或清理失败时停止并重写假设；只有流畅度或内存失败在 40/80 两档各两次都复现，且失败可能由文字栅格化方式改善时，才把该失败记录为 Phase 1 的唯一修复目标。失败是否成立仍不明确时返回决策价值 Gate，不用第二候选补判。

## Phase 1：条件性第二候选

实现 bitmap-backed `CALayer`，只在 40 events/s 目标负载和 80 events/s 余量负载各运行两次，检查是否修复 Phase 0 已确认的唯一失败；不重新运行或重新授权已淘汰的 A。

B 还需通过一次 2× playback rate、pause、双向 seek 和 close 序列。只有它在 40/80 两档各两次都修复 Phase 0 失败、其他流畅度/内存证据不回退，并且没有引入新的语义/清理失败时才选择 B；否则停止并返回决策价值 Gate。

## Phase 2：一次只解决一个不确定性

仅当候选接近、配对结果矛盾、工具明显干扰、生命周期偶发失败，或难回滚生产选择确实依赖该结果时提出预算修订。任何新增 seed、surface、soak、校准、总时限或其他实验维度都必须先返回决策价值 Gate 并重新确认预算；不得恢复旧 broker、PGID、schema/hash、全矩阵或通用 watchdog。

## 实施组件预算

- 一个 executable/process；一个 host/入口；一个固定输入 fixture；一个 recorder；一个 runner。
- 结果使用人可读表格或简单行记录，不定义版本化 schema。
- Phase 0 不创建等待队列或跨事件缓存。fixture 按时间戳与 8 秒可见时长算出最大并发数；候选持有的 layer 总数不得超过该值加一个 admission 临时量。B 使用 BGRA8 backing，单对象最多 163,840 bytes；运行前以“峰值并发加一”推导总 backing 上限，超过 128 MiB 的用户负载拒绝运行并返回决策价值 Gate。回收 layer 时必须清空 bitmap contents。
- Phase 0 不使用 fullscreen、ScreenCaptureKit、外部 supervisor、IPC 或多时钟校准。

## 尚未确定

- 用户已明确确认 D3 的 Phase 0 A 与条件性 Phase 1 B；T3 的技术细节由独立技术审查负责，无需用户逐项复述。实际执行仍等待治理热修合入并新建干净 spike worktree；Phase 2 未授权。
- 当前 Mac 的结果不能外推为最低支持设备或产品上限；这些在生产契约中另行决定。
- 30 分钟、resize/fullscreen、真实播放 surface 和完整资源 Gate 属于胜出路线的生产验证，不由首轮 spike 宣称通过。
