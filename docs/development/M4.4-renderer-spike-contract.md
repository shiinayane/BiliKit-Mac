# M4.4 renderer 不可合入 spike 契约

> 状态：已由用户于 2026-07-22 确认；允许进入不可合入 spike 阶段，尚未确认生产实现契约

## Goal

用同一套全合成、可重复的输入比较 Core Animation 文本 layer 与可复用 AppKit view 两条 renderer 路线，找出各自的容量拐点、资源成本和生命周期失败边界，为后续 M4.4 生产契约提供测量依据。

本 spike 只回答“哪条最小路线值得进入生产设计，以及产品上限应落在哪个已测安全区间”。它不交付可合入 renderer，也不证明所有设备、视频或真实弹幕密度下的性能。

## Context

- M4.3 已提供 `DanmakuEvent → DanmakuScheduler → DanmakuBatch` 数据与调度链路；scheduler 只消费统一媒体时间轴，暂停不继续发射，seek/discontinuity 会要求清空 renderer 语义。
- ADR 0007 将 `BiliDanmaku` 定义为播放呈现 adapter。它只能依赖 `BiliApplication`、`BiliModels` 与必要系统渲染框架；App/Platform 负责把 overlay 组合到播放 surface。
- M4.3 当前最多并发加载 2 段、保留 3 段，batch stream 使用 `bufferingNewest(4)`；这些已确认的数据层上限不是 renderer 的容量结论。
- 当前没有 renderer 的活动对象、等待队列、对象池、帧率或 physical footprint 基线，不能在 spike 前把任何实验数字写成产品承诺。

## 假设

### H1：两条候选路线

- **A — `CATextLayer` pool**：每个活动对象只拥有一个复用的 `CATextLayer`；滚动运动由 layer 上的同一类 Core Animation transform animation 驱动。
- **B — `NSTextField` pool**：每个活动对象只拥有一个 layer-backed、不可编辑且不可选择的 `NSTextField`；滚动运动由其 backing layer 上与 A 相同参数的 Core Animation transform animation 驱动。

唯一允许不同的是文字承载对象为 `CATextLayer` 或 `NSTextField`，以及该对象必需的直接生命周期成本。两条路线必须共享相同的事件生成器、`DanmakuBatch` 输入、lane allocator、文字测量函数、媒体时间控制器、Core Animation 参数、统计口径、实验安全上限和运行矩阵。不得为某一候选单独降低文字质量、跳过模式或放宽清理规则。

共同视觉参数固定为：系统字体 24 pt semibold、sRGB 白色、不加阴影/描边/背景，按当前 surface 的 `backingScaleFactor` 设置内容 scale，并把最终 frame 对齐到物理像素。两者使用同一个 `NSAttributedString` 和同一个 Core Text bounding function；若两条路径无法在 1 个物理像素内得到相同 bounds，该轮标记为不可比较。

滚动对象的可见媒体时长固定为 8 s，顶部/底部对象固定为 4 s。候选都使用同一个媒体时间控制器设置 Core Animation 的 begin time、speed 与 time offset；不得让 B 改用 display link 手动逐帧更新 frame。

### H2：先淘汰语义错误路线，再比较容量

如果候选不能满足本契约“语义、稳定与 quiescence 判定”中的暂停冻结、倍速、seek/discontinuity、identity 替换、surface 替换和关闭清理，即使短时帧率更高也不能进入容量比较的最终候选。

### H3：产品阈值来自重复测量的安全区间

spike 不预设最终最大同屏数、队列深度、对象池大小、丢弃顺序、目标帧率或 physical footprint 增量。测量应找出首次持续退化或资源不再回收的负载拐点；后续生产契约只能选择低于已复现拐点、且留有余量的区间。

## Constraints

### 架构与数据

- 实验输入只能是固定 seed 生成的自制 `DanmakuEvent`；不访问网络、Keychain、真实 BVID/CID、字幕、弹幕正文或账号状态。
- protobuf、endpoint DTO、Cookie、完整 URL、AVPlayer/KVO 类型不得进入 renderer。
- spike 使用假的媒体时间源和合成 surface，但必须通过现有 `DanmakuScheduler.consume(_:)` 取得 `DanmakuBatch`，再把 batch 交给候选 renderer；不得直接把裸 `DanmakuEvent` 喂给 renderer，也不得改变 scheduler 的生产语义来迎合 renderer。
- 只验证滚动、顶部、底部三种基础模式；高级/互动弹幕、直播、ASS、图片/贴纸、描边特效和持久化不在范围内。

### 线程、所有权与清理

- renderer surface 及 AppKit/Core Animation 对象只在主线程或 `@MainActor` 访问；后台任务不得直接持有或变更 UI 对象。
- renderer 必须有单一 owner；callback、display link、Task、animation completion 和观察 token 都必须能追溯到 owner，并在 stop/替换/销毁时解除。所有异步入口在提交 UI 变更前必须再次检查 identity、generation 与 owner 状态。
- 每个提交的 batch 都携带 identity 与 discontinuity generation。旧 identity/generation 的事件、完成回调或复用对象不得重新进入当前 surface。
- pause 期间屏幕位置保持不变；恢复和倍速使用媒体时间继续，不以独立 wall-clock timer 重新计时。
- seek/discontinuity、切换视频/分 P、替换 surface、关闭详情、stop、取消和登出必须有显式清理入口。seek 可以重建新时间窗，但不能继续显示跨越区间的旧对象。
- 清理顺序固定为：先推进 generation 并关闭新 admission，再断开 batch/timeline 入口并取消任务，然后移除 completion/animation/display link，最后回收活动对象和 surface 子对象。`stop` 必须幂等，并通过 `async` quiescence barrier 等待主线程队列和在途 completion 做完最后一次 generation 检查后才返回。
- 状态至少区分 `idle/running/paused/stopping/stopped/closed`；`closed` 不可重新启动。pause 可保留活动对象和池，stop 可保留仅由同一 owner 持有的空闲池供重启，但 active/queue/in-flight callback/Task/display link/animation 必须为零。surface 替换完成后，harness 释放旧 surface 的最后一个外部强引用；其 subview/sublayer/animation/内部强引用必须为零。close 返回后，harness 释放 owner/surface 的最后一个外部强引用；再 drain 两次主线程队列后，包括池和文字 backing resource 在内的 renderer 资源必须全部为零。

### 隐私与证据

- 日志和结果文件只记录候选、环境、seed、输入档位、计数、时序、帧率、内存和清理状态；不记录生成文本本身、事件 ID 列表或任何真实观看内容。
- 原始 Instruments trace、截图或录屏默认不进入 Git。若需要人工复查，只使用合成占位文本，并在结果记录中写明本地路径和保留边界。
- spike 不以单次峰值或一张截图下结论；所有比较必须在同一设备、显示器、窗口状态、构建配置和测量工具下运行。

## 合成输入

事件生成器使用预先公布的 seed 集合 `44/4404/44044`，并在两候选之间复用完全相同的事件序列。PRNG 固定为 SplitMix64 v1，所有运算按 UInt64 overflow 取模：

```text
state += 0x9E3779B97F4A7C15
z = state
z = (z ^ (z >> 30)) * 0xBF58476D1CE4E5B9
z = (z ^ (z >> 27)) * 0x94D049BB133111EB
return z ^ (z >> 31)
```

seed 44 的前四个输出必须依次为 `fb452912299a5453`、`90f83af9167b6122`、`6394fb00725dccd5`、`c8031cf7ec10e77e`。离散选择使用 `next() % upperBound`；bucket 内时间为 `next() % 1_000_000` 微秒，按 `(timeMicros, eventSequenceInBucket)` 升序，其中 `eventSequenceInBucket` 是每个 bucket 内从 0 开始的生成序号，与 scheduler discontinuity generation 无关。manifest 必须预先保存每个 profile/seed 的事件序列 SHA-256，runner 在任何候选启动前重算并匹配，不能在运行时回写期望值。媒体时间以 30 Hz 快照推进，batch 的 identity、generation 与 `clearsExisting` 只来自现有 scheduler：

| 维度 | 固定矩阵 |
| --- | --- |
| 输入速率 | 1、5、10、20、40、80 events/s；每档先预热 30 s，再测量 120 s |
| 基础类型 | scrolling/top/bottom = 80%/10%/10% |
| 文字长度 | 8、24、64 个 grapheme cluster，各占约 60%/30%/10% |
| 文字集合 | 自写 ASCII、CJK、emoji/组合字符占位集合；每轮按相同索引循环，不得取自真实弹幕 |
| surface | 640×360、1280×720、1920×1080 points；另测当前显示器全屏 |
| 播放状态 | 1×、pause 10 s、恢复、2× 20 s、向前 seek 60 s、向后 seek 30 s |
| surface 变化 | 1280×720 ↔ 640×360 连续 10 次、进入/退出全屏 3 次、替换承载 surface 10 次 |
| 生命周期 | identity/generation 替换 20 次、stop/start 20 次、关闭后观察回收 60 s |

运行矩阵分三层：

1. **语义矩阵**：固定 5 events/s。对 `candidate × seed × surface` 做完整笛卡尔积；每轮在同一事件序列中依次执行 1×、pause、恢复、2×、双向 seek、10 次 resize、3 次 fullscreen 往返、10 次 surface 替换、20 次 identity/generation 替换、20 次 stop/start，最后 close。普通 window surface 分别冷启动；fullscreen 使用当前显示器且不与其他尺寸同时组合。
2. **负载阶梯**：固定 1280×720 和纯 1× 连续播放，对 `candidate × seed × 1/5/10/20/40/80 events/s` 展开；每档独立冷启动。完成同一档的三个 seed 后，若按“档位稳定”规则判定为不稳定，停止该候选的更高档，避免把已知失败继续升级负载。
3. **候选 soak**：每条仍满足语义不变量的候选运行 30 分钟。若所有档位稳定，soak 80 与 40；若至少两个档位稳定，soak 最高稳定档和紧邻的更低稳定档；若只有 1 events/s 稳定，只 soak 1；若 1 events/s 不稳定，该候选直接淘汰、不运行 soak。

每个非 soak 组合以三个 seed 各运行一次；候选按 AB/BA 平衡顺序成对执行，关键拐点再完整复跑一组，检查温度、缓存或候选顺序偏差。记录当前 Mac 型号、芯片、内存、macOS、Xcode/Swift、显示器分辨率/刷新率/`backingScaleFactor`/色彩空间、appearance、窗口可见/遮挡状态、供电/低电量/thermal state、测量工具版本与精度和 Release 构建标识，不把本机结果外推为最低支持设备结论。

## 实验安全上限

以下数字只防止错误实验失控，不是产品默认值或验收承诺：

- `offered` 是生成器在整轮产生的累计数量，不设 4,096 上限；`pendingIngressEvents` 是已经 offer 但尚未 admission/drop/expire 的瞬时事件集合，硬上限为 4,096。输入按媒体时间 lazy delivery，只有即将进入当前 30 Hz bucket 的事件才进入该集合。
- `queuedEvents` 是已经 admission、但等待 lane/对象的事件集合，硬上限为 4,096；事件从 `pendingIngressEvents` 只能一次性转移到 active、queued、dropped 或 expired，不能同时属于两个集合。queued event 不拥有 renderer object。
- `allocatedRenderObjects = activeObjects + pooledObjects + inFlightRecycleObjects`，三个对象集合互斥，总硬上限 1,536；其中 active 上限 512、pool 上限 1,024、in-flight recycle 上限 512。allocation 必须在主线程确认 pool 无可复用对象且总量仍小于 1,536 后才发生，禁止先分配后补记计数。
- 队列满时按固定规则立即 drop 新到达事件并记录 `queue-capacity`；活动或池上限只触发复用/排队/丢弃，不允许越界创建。累计输入继续运行，以观察确定性背压而不是提前截断高负载档。
- 相对该轮空 surface idle 基线，physical footprint 增量达到 1 GiB、进程崩溃或系统内存压力警告时终止该轮。主线程每 250 ms 只 ack 一个内存 counter，不执行文件 I/O；后台 reporter 读取 counter 并原子写 heartbeat。独立 watchdog 若 5 s 未观察到主线程 ack 推进，则先保存外部状态并发送 `TERM`，2 s 后仍未退出才发送 `KILL`。
- 安全上限触发只说明该轮无效或候选在该负载不可用，不能直接成为未来产品阈值。

每一轮必须在全新子进程中运行。runner 优先通过 `posix_spawn` 的 `POSIX_SPAWN_SETPGROUP` 属性在 spawn 时建立独立进程组；若当前 SDK 无法提供该路径，只允许 child 在 exec 前调用 `setpgid(0, 0)` 并与 parent 完成握手，不能由 parent 在 child exec 后竞态设置。parent 随后用 `getpgid` 验证 PGID 等于 childPID、且不等于 runner PID/PGID 或任何其他轮次 PGID；验证失败时以 harness 错误停止，绝不发送 group signal。watchdog 只能向这一个已验证 PGID 发信号。runner 在开始下一轮前确认旧进程组不存在、临时 surface 已关闭，并等待 thermal state 回到该组运行前等级或最多 5 分钟；超过后将后续结果标记为 thermal-contaminated，不纳入 A/B 选择。

## 语义、稳定与 quiescence 判定

以下是 spike 的预注册判定规则，不是产品性能承诺：

- **语义通过**：旧 identity/generation 再次可见、重复呈现、lane 碰撞和 surface 越界计数必须为 0；pause 发出后等待两个实际刷新周期，随后 10 s 内 presentation position 变化不得超过 1 个物理像素。事件媒体时间到 scheduler batch 的误差单独记录，必须落在 `[0, oneTimelineTick + clockCalibrationError]`；batch 到首次 presentation、以及已呈现对象相对其 8 s/4 s 媒体生命周期的退出误差，分别不得超过 `twoDisplayPeriods + clockCalibrationError`。整体进入误差上限是两段上限之和，不能把 30 Hz scheduler 量化归因给 renderer。
- **清理通过**：收到 `clearsExisting` 后两个刷新周期内旧活动对象、旧 animation 和旧 completion 为 0。`stop` 的 quiescence barrier 返回时 active/queue/in-flight callback/Task/display link/animation 为 0；harness 释放最后一个外部强引用后，旧 surface 立刻没有 subview/sublayer/animation/内部强引用。`close` 返回、harness 释放 owner/surface 的最后一个外部强引用并再 drain 两次主线程队列后，pool、文字 backing resource 和 owner/surface 弱引用也必须为 0。
- **内容推进通过**：每次显示回调从当前活动滚动对象中按最低 event sequence 选择最多 8 个 sentinel，并读取 presentation layer 的实际 x。排除状态切换前后两个显示周期；当媒体时间预计位移累计至少 1 个物理像素时，实际位移小于 0.5 像素记为 stalled sample，实际位置偏离媒体时间理论位置超过 `max(2 physical pixels, twoDisplayPeriodsExpectedDistance)` 记为 jump sample。eligible callback 中 presentation sample 有效率必须至少 95%，stalled+jump 不多于 1%，且连续 stalled/jump 不得超过 2 个 callback；否则即使显示回调本身准时，该轮也不稳定。无法读取 presentation 或无法建立理论位置时，该轮测量无效。
- **单轮稳定**：语义、内容推进和清理通过，未触发安全上限；测量窗口内 frame interval p95 不超过两个实际刷新周期，超过两个刷新周期的帧不多于有效帧的 1%；最后 60 s 的 active/queue/pool 各自最大值不高于前 60 s 最大值，且 120 s 窗口的 physical-footprint Theil–Sen slope 不大于 `max(1 MiB/min, idle physical footprint 的 0.5%/min)`。
- **档位稳定**：三个 seed 的三轮都单轮稳定。任一轮失败则该档位不稳定；不删除 outlier。
- **容量拐点**：按输入速率递增时，第一个不稳定档位；它出现后不把偶然的更高档结果重新定义为稳定区间。若 80 events/s 仍稳定，只能报告“本矩阵内未观察到拐点”，不得外推更高容量。
- **最高稳定档位**：容量拐点之前最近的稳定档位；最低档已不稳定时不存在最高稳定档。soak 按上方边界规则选择，不把结果反向修改上述判定公式。

physical-footprint 斜率只用于发现单调增长风险，不等同于泄漏证明；即使该判定通过，资源计数或 close 归零失败仍直接淘汰候选。

## 测量项

每 1 s 聚合一次，结果以不含正文的 JSON Lines 保存；runner 另写一份 summary JSON。基准单调时间来自 `ContinuousClock`，frame interval 来自同一 surface 的显示回调时间戳，process physical footprint 来自 `TASK_VM_INFO.phys_footprint`。每轮开始和结束时用同一次主线程 transaction 同时采样 ContinuousClock、假媒体时间、`CACurrentMediaTime()`/layer local time 和显示回调时间戳，记录四者 offset、最大 round-trip jitter 与 `clockCalibrationError`；校准误差超过一个实际刷新周期时该轮测量无效。若实际 API 或工具不可用，必须在开工前回到本契约改口径，不能静默替换。

### 输入、调度与降级

- offered、admitted、rendered、queued、dropped、expired 事件数；
- 按 scrolling/top/bottom 和丢弃原因分组的计数；
- 当前/峰值活动对象、等待队列、对象池、累计创建、复用、回收和跨轮残留数；
- lane 使用数、无可用 lane 次数、已检测碰撞/越界/重复呈现数；
- batch 接收至首次可见的延迟分布，但不保存事件标识。

### 帧与主线程

- 实际显示器刷新率、采样帧数、frame interval 的 p50/p95/p99、超过一个和两个刷新周期的帧比例；
- sentinel presentation sample 的 eligible/valid/stalled/jump 计数、最大连续异常 callback 数和相对媒体时间位置误差分布；
- 主线程 CPU 时间、renderer 回调耗时分布，以及动画提交次数；
- 若测量工具不能区分 App、WindowServer 与 GPU 成本，结果必须明确标为“不可归因”，不得用单一进程 CPU 替代总渲染成本。

### 内存与文字资源

- 空 surface idle physical footprint、每档峰值 physical footprint、档位结束值、清理后 10/30/60 s 值；
- layer/view、animation、文字 backing store 或等价文字资源的创建、峰值、复用和清理计数；
- 连续档位或 soak 中 physical footprint/对象数随时间的趋势斜率；只报告测量区间与误差，不把一次尾部值写成“无泄漏”。

### 时间轴与生命周期正确性

- pause 10 s 期间的最大位置变化；
- 1×/2× 下以媒体时间计算的进入、退出误差；
- seek/discontinuity 后旧对象、旧 completion 和旧 generation 再次可见的数量；
- resize/fullscreen/surface 替换后的越界、重复喷射和错误 lane 数；
- stop、identity 替换和关闭后剩余 callback、Task、observer/display-link、animation、活动对象、队列、池和文字 backing resource 计数，以及 owner/surface 弱引用是否释放。

## 运行入口与结果 schema

spike 分支必须提供唯一入口，并为 CLI 参数、退出码、manifest JSON Schema 和结果 JSON Schema 建立 contract test；`sh -n` 只额外检查 shell 语法。参数或 schema 变化必须回到契约审查：

```sh
DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer \
  sh Scripts/run-m44-renderer-spike.sh \
  --candidate cat-text-layer,nstext-field \
  --manifest Spike/M44Renderer/input-v1.json \
  --results /tmp/BiliKitMac-m44-renderer/<run-id>
```

runner 在任何候选子进程启动前必须先自动执行同一入口的 `--self-test` 模式；也允许独立复查：

```sh
DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer \
  sh Scripts/run-m44-renderer-spike.sh \
  --self-test \
  --manifest Spike/M44Renderer/input-v1.json \
  --results /tmp/BiliKitMac-m44-renderer/self-test-<run-id>
```

`--self-test` 必须覆盖 CLI/schema contract、manifest/hash、生成器、计数器、时钟校准、空/静态基线和本契约列出的全部负向注入；任一检查失败时 runner 以 50 退出且不得启动候选。正常模式的 summary 必须引用本次 self-test runID 和结果 hash；缺少或不匹配时整组测量无效。

self-test 后、语义矩阵或负载阶梯前，runner 必须以三类合成文字运行一次 `visual-preflight`，为 A/B 各生成静态截图并暂停。人工检查物理像素尺寸、截断、baseline 和可读性后，将 pass/fail、检查时间和截图 hash 写入 preflight summary；截图本身不进入 Git。任一类型存在肉眼可见差异时整对以 40 退出，不运行后续候选矩阵。

`input-v1.json` 固定本节全部 seed、分布、视觉参数、时间轴、surface 和生命周期矩阵；runner 在读取后计算其 SHA-256 并写入结果。runner 为每轮生成不可重复 `runID`，每个结果至少包含：schemaVersion、runID、candidate、spike commit、input manifest hash、seed、环境字段、工具版本、开始/结束 monotonic time、exit/stop reason、heartbeat/watchdog 结果、每秒聚合、语义判定、资源 quiescence 和 summary。两候选的输入摘要 hash 不同则整对作废。

runner 退出码固定为：0=该轮完成且结果可判定，10=候选语义/稳定失败，20=安全停止，30=watchdog kill，40=测量无效，50=harness/manifest 错误。即使非零退出，也必须由 runner 在进程外保存 manifest hash、最后一个完整聚合、退出原因、quiescence 可观测结果和工具版本；本地 raw trace 最长保留 7 天且不进入 Git。

## 必须先实现的测量可信度检查

- 事件生成器必须匹配 manifest 中预注册的每个 profile/seed 摘要 hash；两候选每轮还须记录相同输入摘要。
- 对计数器注入已知数量的创建、复用、丢弃和清理操作，确认聚合结果精确一致。
- 空 surface 与无动画静态文字基线分别测量，并执行与负载轮次相同的最多 8 个 sentinel presentation 查询，量化观察者开销并区分 harness 自身成本和 renderer 成本。
- `visual-preflight` 必须在负载测量前完成；截图只用于上述人工质量检查，不作为性能数据。
- 人工制造旧 generation completion、off-main callback、stop 与 batch 提交竞态、取消后 completion、复用对象旧 completion、surface 替换期间 animation completion、未显式 stop 的 owner 销毁、未解除 callback 和一次 allocation-before-check 越界，确认 probe 对每种情况都能失败；不能只验证成功路径。
- measurement build 固定为同一 Release 配置，关闭调试日志；任何候选若需要不同编译选项，必须说明并单独标记不可直接比较。

## 立即停止条件

出现以下任一情况，停止当前候选或整个 spike，不继续用更多调参掩盖：

- 无法让两候选消费相同输入或得到可比较的时钟、内容推进、帧与 physical-footprint 口径；
- crash、hang、安全上限触发、主线程违规或资源 owner 无法说明；
- pause/seek/discontinuity/identity/surface 清理语义失败，且最小修正会改变生产 scheduler、公共 API 或架构边界；
- stop quiescence 后仍有旧 callback、Task、observer/display-link、animation、活动渲染对象或等待队列；close 后仍有池、文字 backing resource、owner/surface 强引用；
- 需要真实网络/个人数据、第三方 renderer、私有 API、额外 Package 或产品代码改动才能继续；
- 同一候选同一路线连续两次修正仍无法取得可信测量，或两个 reviewer 对资源/线程结论冲突。

停止后只能得出“候选被淘汰”“测量无效”或“需要新契约”，不能把未完成矩阵解释为性能通过。最小脱敏证据必须保留 input manifest hash、最后一个完整聚合、停止原因、quiescence 检查、工具版本和本地 trace 到期时间。

## Done when

spike 只有在以下条件全部满足后才算关闭：

1. 用户已在实验前确认本契约，运行记录包含所用 commit、环境和明确的不可合入标记。
2. 测量可信度检查通过；所有有效候选完成语义矩阵、3 次负载阶梯和规定的 30 分钟 soak，或按停止条件留下可复查的淘汰证据。
3. 结果明确给出每条候选的容量拐点区间、内容推进/帧/physical-footprint/文字资源趋势、清理结果、已知噪声与不可外推边界；不以“感觉流畅”代替数据。
4. 独立上下文分别完成失败/过载场景审查，以及线程、所有权、generation、安全上限和清理路径的 `red_reviewer` 只读终审；结论按 blocker/improvement/reject 记录。
5. spike worktree 已关闭，实验产物未合入生产分支；生产代码、公开 API、Gate 和历史验证记录没有被实验改写。
6. 基于数据另起一份待用户确认的 M4.4 生产契约。它至少固定活动对象/队列/对象池上限、lane/碰撞/丢弃规则、暂停/倍速/seek/resize/fullscreen 行为、旧 generation 清理和生产 Gate。

## spike 关闭时允许的结论

只能选择以下一种：

- **进入生产契约**：至少一条候选满足语义和清理不变量，并形成有重复测量支持的安全区间；
- **淘汰全部候选**：两条候选均触发停止条件，需要先重写假设和契约；
- **测量无效**：工具或 harness 无法提供可信比较，不能选择 renderer 路线。

## 用户确认项

开始 spike 前，用户需要明确确认：

- [x] 接受 A/B 两条候选和“先语义、后容量”的比较顺序；
- [x] 接受固定的三 seed、完整语义矩阵、负载阶梯、预注册判定公式和 30 分钟 soak 边界；
- [x] 接受实验安全上限不等于产品阈值；
- [x] 接受每轮独立子进程、主线程 ack heartbeat 和经校验 PGID 的外部 watchdog 终止路径；
- [x] 接受 spike 分支及其中所有实现不可合入；
- [x] 接受本轮不触碰生产 renderer、App composition、真实网络、持久化或 UI 精修；
- [x] 接受 spike 结束后再次确认生产契约，才开始 M4.4 正式实现。
