# M4.5 视频卡片视觉复用决策

> Revision：C3。状态：已完成。2026-07-24 用户确认观看历史真实视觉并同意结束 Slice A；三路前置审查、实现后独立复核、Package／App Gate 与签名 App 观察均通过。

## 为什么做

热门和搜索已经形成获用户认可的 16:9 卡片视觉，但观看历史仍是另一套列表行。继续复制圆角、封面比例、标题字号、头像和间距，会让三个页面在后续微调时逐渐分叉。

本轮只决定两件事：

1. 怎样让 Browse 与 Library 两个 Feature 共用同一套视频卡片基础视觉，同时保持 Feature 之间不互相依赖。
2. 怎样让观看历史改为相同卡片流，并只替换它真正不同的信息。

## 选择

在现有 `BiliKitCore` Package 内新增一个仅供 Package 内部使用的窄 `BiliUI` target，不新增公开 library product。它只依赖 SwiftUI／Foundation，不依赖 `BiliModels`、`BiliApplication` 或任何 Feature。`BiliBrowseFeature` 与 `BiliLibraryFeature` 都只向它传入已经格式化好的显示文本和准备完成的图片 URL。长期边界由 [ADR 0009](../adr/0009-narrow-biliui-video-card-boundary.md) 固定。

共享卡片固定以下视觉：

- 16:9 封面、圆角、选中描边和底部渐变；
- 标题字号、最多两行及保留的两行高度；
- 可选 UP 主头像、底部元数据字体和间距；
- 与首页相同的响应式卡片网格。

共享卡片允许省略头像、封面左侧指标、封面右侧文本或底部右侧文本。省略后不保留空白占位，也不伪造服务端没有提供的信息。

“是否显示头像槽位”和“头像 URL 是否存在”是两件事：热门／搜索继续显示头像槽位，URL 缺失时使用现有非业务占位图；历史仅在 API 确实提供头像 URL 时显示该槽位。

各 Feature 继续决定业务文本：

- 热门／搜索：封面显示播放量、弹幕数和总时长；标题下显示 UP 主与发布日期。
- 观看历史：不显示播放量和弹幕数；封面显示 `已看时长/总时长`，明确看完时显示“已看完”；标题下左侧显示 UP 主，右侧按“今天／昨天 `HH:mm`，更早 `M月d日 HH:mm`”显示观看时间。

## 为什么不直接复用 `GuestVideoCard`

`GuestVideoCard` 属于 `BiliBrowseFeature`。让历史页面 import Browse 会破坏 Feature 独立边界；复制一份则无法保证视觉持续一致。把整个领域模型或历史语义移入 UI target 又会让基础 UI 反向拥有业务规则。因此只提取无业务模型的视觉外壳。

## 不做什么

- 不修改历史 API、DTO、Application port 或领域模型。
- 不增加图片缓存、预解码、性能框架或第三方依赖。
- 不建立通用设计系统、主题系统、任意 View registry 或 `Shared/Utils`。
- 不改变分页、认证、选择视频、返回和播放器生命周期。
- 不为“已看完”猜测 90% 等产品阈值；只使用现有 adapter 已归一化的 `progress >= duration` 明确信号。

## 最低证据与停止条件

完成至少需要：

- 热门、搜索保持当前视觉与选择行为；历史使用同一基础卡片流。
- 历史卡片正确覆盖进行中、已看完、无头像和跨小时总时长。
- 分页、空状态、失败、重试、刷新禁用、退出清理和选择播放的既有测试保持通过。
- 架构脚本明确允许 Feature 依赖 `BiliUI`，同时阻止 `BiliUI` 依赖领域、Feature 或 adapter。
- 卡片的文字指标和选中状态可被 VoiceOver 感知；封面图片、渐变等装饰元素不重复朗读。
- Package 与 App Gate 通过，并由用户观察历史页真实视觉。

如果共享组件需要理解 `PopularVideo`、`SearchVideo` 或 `WatchHistoryItem`，需要引入类型擦除／环境注册，或历史布局无法在同一卡片结构内自然表达，则停止复用并返回决策 Gate。

## 复杂度预算

- 只新增一个 Package 内部 `BiliUI` target，不新增 Package、library product、App framework product 或第三方依赖。
- `BiliUI` 首轮最多三个生产类型：视频卡片、封面指标值、网格布局。
- 三个类型及其跨 target 成员使用 Swift `package` 访问级别；工程契约禁止增加 `BiliUI` library product。
- 不新增 `BiliUITests` 或像素快照框架；复用既有 Feature 测试、架构检查与真实 App 观察。
- 不新增缓存、actor、Task、runner、fixture、schema、probe 或 benchmark。
- 不移动网络、认证、播放或资源生命周期 owner。
- 超出以上任一项时停止，并说明新增成本保护了哪个实际决策后重新请求用户确认。
