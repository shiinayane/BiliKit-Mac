# M4.4 renderer 生产决策契约

> Revision：P4。状态：用户已于 2026-07-23 确认当前 100% 显示区域的真实视觉效果，并进一步澄清覆盖比例采用上下镜像锚定：顶部/滚动从上缘覆盖，底部从下缘覆盖。显示区域和密度设置仍不进入本轮 UI。

## 通俗摘要

v1 会在现有播放器上显示滚动、顶部和底部三类基础弹幕，采用 spike 已验证、较流畅的文字绘制方式，避免逐条阴影造成的严重卡顿。弹幕必须跟随暂停、倍速、跳转和视频切换；无法安全显示或容量已满时跳过新弹幕，不延迟堆积。v1 只做一个简单实现，同时隔离具体绘制方式，真实证据不足时再评估更复杂路线。

## 本轮交付

- 在现有视频 surface 上组合滚动、顶部和底部弹幕 overlay，提供总开关和三类过滤。
- v1 使用 spike 选出的 `CATextLayer + 内容级重墨`，禁止恢复逐 layer compositor shadow。
- 复用 M4.3 的 `DanmakuSession`、统一播放时间轴、过滤和 generation，不增加独立计时源。
- pause 冻结当前位置，倍速同步改变运动；seek/discontinuity、切换视频/分 P、关闭页面会清空旧内容，旧回调不得进入新状态。
- resize/fullscreen 后重新建立轨道，不回填已经跳过的历史弹幕。
- 三类弹幕共享同一个覆盖比例：顶部与滚动从上缘向下覆盖，底部从下缘向上镜像覆盖。覆盖范围相交时，不同类型可以占用相同位置；当前默认只防止同类弹幕互相碰撞。
- 同类没有安全 lane 或达到有界上限时丢弃尚未显示的新弹幕，不建立延迟队列，也不中途移除已经上屏的弹幕。

密度、显示区域、字号、透明度和偏好持久化不进入本轮 UI；但底层模型必须保持正确扩展方向：未来 50% 表示顶部/滚动使用上方 50%、底部使用下方 50%；75% 的上下区域会在中间重叠；100% 时三类都覆盖完整 surface。未来“正常／较多／重叠”调整同类碰撞和密度策略。

## 架构与未来替换

- `BiliDanmaku` 拥有 lane/碰撞、呈现命令和 renderer 生命周期；Feature 只表达总开关和三类过滤。
- App/Platform 负责挂载具体 overlay，Composition 是连接数据、时间轴、renderer 和播放 surface 的唯一位置。
- scheduler 与 Feature 不暴露具体 layer 类型。v1 只实现一个 Core Animation backend，但未来可在不改调度、generation 和用户意图的前提下替换挂载 adapter 与 renderer。

## 不做什么

- 不支持高级/互动/直播弹幕、发送、ASS 导出或下载。
- 不实现 45° 投影、第二 renderer、Metal、glyph atlas、通用缓存或 benchmark framework。
- 不把 spike runner、合成 host、实验目录或 Instruments 记录合入生产。
- 不用当前 Mac 的 spike 结果承诺所有设备的容量。

## 完成与停止

只有以下证据同时成立才完成：

- lane/碰撞、有界丢弃、pause、倍速、seek/generation、resize 和清理的确定性测试通过；
- Package、App 与架构 Gate 通过；
- 当前 Mac 的真实播放器上完成三类弹幕、resize/fullscreen、切换视频和关闭页面验证；
- 30 分钟受控运行没有明显卡顿、无界资源增长或清理失败。

若生产集成重现严重卡顿、资源不能归零，或必须增加第二 renderer、通用框架才能完成，则停止并返回决策价值 Gate。

## 复杂度预算与下一步

- 一个 renderer backend、一个 overlay surface、一个 lane allocator。
- 不新增 Swift Package；不预建 bitmap cache、layer pool、后台栅格化队列或第二时间轴。
- 超出预算时，必须说明新增成本保护了哪个 v1 决策，并重新请求用户确认。
- P4 的镜像覆盖模型通过确定性测试、App Gate 和独立红区审查后，才恢复 30 分钟收口验证；当前 100% 显示区域的真实视觉效果已经用户确认。

## 可理解性 Gate

用户明确同意且没有影响授权的重大误解即视为通过；无需复述技术细节：

1. 是否同意 v1 采用已验证、较流畅的文字绘制方式，不使用会造成严重卡顿的旧阴影效果？
2. 是否同意弹幕无法安全显示或容量已满时直接跳过新弹幕，以避免延迟和堆积？
3. 是否同意 v1 先保持一个简单实现，只有真实使用证据表明不足时再评估更复杂的绘制方案？
